# AI 模型量化技术 | GGUF / AWQ / GPTQ 全解析

大模型动辄几十 GB，如何压缩到消费级设备可运行？量化技术就是答案。

## 一、什么是模型量化

量化的本质是将高精度浮点数映射到低精度表示。

```
FP32（32位浮点数） → FP16（半精度）→ INT8 → INT4
        模型大小: 100% → 50% → 25% → 12.5%（理论值）

实际 7B 模型大小：
FP32：~28GB
FP16/BF16：~14GB
INT8：~7GB
INT4：~3.5GB
```

### 量化为什么能工作
- 神经网络权重分布通常集中在较窄的范围内
- 推理不需要全精度的"微妙差异"
- 适当的缩放因子可以保留关键信息

## 二、主流量化方法对比

| 方法 | 精度 | 模型大小 | 推理速度 | 效果损失 |
|------|------|---------|---------|---------|
| FP16 | 无损 | 50% | 快 | 无 |
| INT8 | 微小 | 25% | 很快 | 几乎无 |
| GPTQ INT4 | 少量 | 12.5% | 快 | 少量 |
| AWQ INT4 | 少量 | 12.5% | 最快 | 少量 |
| GGUF Q4_K_M | 少 | ~4GB | 中 | 较小 |
| GGUF Q2_K | 较多 | ~2.5GB | 中 | 明显 |

## 三、GPTQ 量化

### 原理
GPTQ 逐层量化权重，每量化一层后对剩余层进行补偿，最小化量化误差。

```python
# 使用 AutoGPTQ
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig

# 量化配置
quantize_config = BaseQuantizeConfig(
    bits=4,               # 4bit 量化
    group_size=128,       # 分组大小
    damp_percent=0.01,
    desc_act=False,       # 是否按激活值排序（True 效果更好但更慢）
)

# 加载并量化模型
model = AutoGPTQForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-7B-Instruct",
    quantize_config
)

# 使用校准数据集量化
model.quantize(calibration_dataset)

# 保存量化模型
model.save_quantized("./qwen-7b-gptq-int4")
```

### 优缺点
- ✅ 压缩率高，4bit 只需 ~4GB
- ✅ GPTQ 格式被 vLLM、TGI 等推理引擎广泛支持
- ❌ 量化过程较慢（需几个小时）
- ❌ 需要校准数据集

## 四、AWQ 量化

### 原理
AWQ 发现只有约 1% 的"显著权重"对精度至关重要，保护这些权重即可。

```python
# 使用 AutoAWQ
from awq import AutoAWQForCausalLM

# 加载模型
model = AutoAWQForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-7B-Instruct",
    safetensors=True
)

# AWQ 量化配置
quant_config = {
    "zero_point": True, 
    "q_group_size": 128, 
    "w_bit": 4, 
    "version": "GEMM"
}

# 量化
model.quantize(
    tokenizer,
    quant_config=quant_config,
    calib_data=calibration_dataset
)

# 保存
model.save_quantized("./qwen-7b-awq-int4")
```

### 优缺点
- ✅ 推理速度比 GPTQ 快
- ✅ 量化过程相对快
- ✅ 效果与 GPTQ 相当或略好
- ❌ 不如 GGUF 灵活（只能固定量化位宽）

## 五、GGUF 格式（llama.cpp）

GGUF 是 llama.cpp 的模型格式，最灵活的选择。

### 量化级别说明
```
Q2_K  → 极低质量，~2GB，仅用于极端资源受限
Q3_K_S/M/L → 低质量，~3GB
Q4_K_S/M → 中等质量，~4GB ← 最常用
Q4_K_M → 常用推荐，效果与速度平衡
Q5_K_S/M → 高质量，~5GB
Q6_K  → 接近无损，~6GB
Q8_0  → 几乎无损，~7GB
F16   → 无损，~14GB
```

### 量化与推理
```bash
# 使用 llama.cpp 量化模型
# 1. 转换 PyTorch 模型为 GGUF
python convert_hf_to_gguf.py Qwen/Qwen2.5-7B-Instruct --outtype f16

# 2. 量化
./llama-quantize model-f16.gguf Q4_K_M

# 3. 推理
./llama-cli -m model-Q4_K_M.gguf -p "你好，请介绍下自己"

# 4. 启动 API 服务
./llama-server -m model-Q4_K_M.gguf -c 4096 --port 8080
```

### Ollama 集成
```bash
# Ollama 内部使用 GGUF
# 创建 Modelfile
FROM ./qwen-7b-q4.gguf

# 创建并运行
ollama create my-model -f Modelfile
ollama run my-model
```

## 六、量化方法选择指南

```
你的场景是？
│
├─ 服务器推理部署
│  ├─ 用 vLLM → AWQ（vLLM 原生支持）
│  └─ 用 TGI → GPTQ
│
├─ 个人电脑本地运行
│  ├─ 用 Ollama → GGUF Q4_K_M
│  └─ 用 llama.cpp → GGUF Q4_K_M
│
├─ 移动端 / 边缘设备
│  └─ GGUF Q3_K_S 或更小
│
└─ 追求极致效果
   └─ 不量化！FP16 部署

```

## 七、量化效果对比实验

```python
# 快速评估量化对效果的影响
def compare_quantization(original_model, quantized_model, test_set):
    """对比原模型和量化模型的效果差异"""
    results = []
    for case in test_set:
        orig_output = original_model.generate(case["input"])
        quant_output = quantized_model.generate(case["input"])
        
        # 计算输出相似度
        from difflib import SequenceMatcher
        similarity = SequenceMatcher(None, orig_output, quant_output).ratio()
        
        results.append({
            "input": case["input"],
            "similarity": similarity,
            "orig_len": len(orig_output),
            "quant_len": len(quant_output)
        })
    
    avg_sim = sum(r["similarity"] for r in results) / len(results)
    print(f"平均输出相似度: {avg_sim:.2%}")
    
    return results
```

## 八、注意事项

| 注意点 | 说明 |
|--------|------|
| 校准数据 | GPTQ/AWQ 需要校准数据，通常 128 条即可 |
| 不可逆 | 量化后无法恢复原始精度 |
| 显存 ≠ 模型大小 | 推理时还需 KV Cache 等额外显存 |
| 微调兼容 | LoRA 可在量化模型上训练（QLoRA） |
| 平台兼容 | GGUF 不适用于 GPU 服务器；GPTQ/AWQ 不适用于 CPU |

## 总结

> 量化是用少量的精度损失，换取巨大的部署灵活性。

**推荐组合：AWQ（服务器）+ GGUF Q4_K_M（个人设备）= 覆盖所有场景。**
