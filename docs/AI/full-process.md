# AI 全流程讲解 | 从想法到落地的完整路径

一个 AI 项目从 idea 到上线，需要经历多个环环相扣的阶段。本文将系统梳理 AI 开发的完整流程，帮助你建立端到端的项目认知。

## 全流程概览

```
需求定义 → 数据准备 → 模型选型 → 训练评估 → 部署上线 → 监控迭代
```

## 一、需求定义

一切始于清晰的问题定义。不是所有问题都需要 AI，也不是所有场景都适合大模型。

**关键问题：**
- 要解决什么业务问题？输入和输出分别是什么？
- 这个问题，传统规则 / 简单统计能不能解决？
- AI 方案的投资回报率（ROI）是否值得？

**输出产物：** 需求文档、评估指标定义、可行性分析报告

## 二、数据准备

数据决定了模型的天花板，算法只是逼近这个天花板。

### 2.1 数据采集
- **公开数据集**：Hugging Face Datasets、Kaggle、OpenDataLab
- **自有数据**：业务日志、用户标注、网络爬取
- **合成数据**：用大模型生成训练数据（Self-Instruct、Evol-Instruct）

### 2.2 数据清洗
- 去除重复、低质量、格式异常样本
- 隐私信息脱敏（手机号、身份证、密钥等）
- 文本长度过滤（太短无意义，太长超出窗口）

### 2.3 数据标注
- **人工标注**：质量最高，成本最大
- **半自动标注**：模型预标注 + 人工复核
- **自动标注**：用更强的模型（如 GPT-4）打标

### 2.4 数据增强
- 同义词替换、回译（Back Translation）
- Prompt 改写多样化
- 少样本 → 多样化生成

### 2.5 数据划分
- **训练集（Train）**：70%-80%，模型学习
- **验证集（Val）**：10%-15%，超参调优
- **测试集（Test）**：10%-15%，最终评估

## 三、模型选型

根据任务类型、资源预算和性能要求选择合适的模型。

| 考量维度 | 说明 |
|---------|------|
| 任务类型 | 文本生成 / 分类 / 抽取 / 代码 / 多模态 |
| 模型规模 | 7B / 13B / 70B，越大效果越好但越贵 |
| 语言能力 | 中文效果（Qwen 系列较好）、多语言需求 |
| 开源协议 | Llama、Qwen、DeepSeek 可商用，Gemma 有限制 |
| 推理成本 | 部署需要多少显存？推理延迟是否可控？ |

**主流开源模型（2025-2026）：**
- **Qwen 系列**（阿里） — 中文能力突出，全尺寸覆盖
- **DeepSeek 系列**（深度求索） — 高性价比，MoE 架构
- **Llama 系列**（Meta） — 生态最丰富，社区活跃
- **Mistral 系列** — 轻量高效

## 四、训练与评估

### 4.1 训练准备
- 环境搭建：GPU 服务器 / 云平台（AutoDL、恒源云）
- 框架选择：LLaMA-Factory、Hugging Face Trainer、Axolotl
- 混合精度训练（bf16/fp16）节省显存
- 梯度累积：小批次模拟大批次

### 4.2 关键超参数
```
学习率（Learning Rate）     → 通常 1e-5 ~ 5e-5
批次大小（Batch Size）       → 根据显存调整，配合梯度累积
训练轮数（Epochs）           → 2-5 轮，过多容易过拟合
warmup 比例                  → 总步数的 3%-10%
权重衰减（Weight Decay）     → 0.01 ~ 0.1
```

### 4.3 评估策略
- **自动评估**：BLEU、ROUGE、Perplexity、准确率
- **人工评估**：A/B 对比、Likert 评分、盲测
- **基准测试**：C-Eval、CMMLU（中文）、MMLU、HumanEval
- **业务评估**：与实际业务指标（转化率、满意度）的关联

### 4.4 常见问题排查
| 现象 | 可能原因 | 解决方向 |
|------|---------|---------|
| 训练 loss 不下降 | 学习率过大/过小 | 调整 lr，warmup |
| 验证集过拟合 | 训练太久 / 数据太少 | 早停、增加正则、数据增强 |
| 模型输出重复 | 解码策略问题 | 调 temperature、top_p，加 repetition_penalty |
| 灾难性遗忘 | 只在领域数据上训练 | 混合通用数据，减小学习率 |

## 五、部署上线

### 5.1 模型导出
- 合并 LoRA 权重到基座模型
- 格式转换：PyTorch → GGUF（llama.cpp）/ ONNX / TensorRT
- 量化压缩：int8 / int4 / GPTQ / AWQ

### 5.2 推理服务
- **vLLM**：高性能推理引擎，PagedAttention
- **Ollama**：一键本地部署，适合个人使用
- **TGI（Text Generation Inference）**：Hugging Face 官方方案
- **SGlang**：结构化生成更友好

### 5.3 API 封装
- FastAPI 包装推理接口
- 流式输出（SSE / WebSocket）
- 速率限制与认证鉴权
- 请求队列与负载均衡

## 六、监控与迭代

模型上线不是终点，持续迭代才是常态。

### 监控指标
- **性能监控**：延迟（P50/P95/P99）、吞吐量、错误率
- **质量监控**：输出分布漂移、拒绝率、平均长度变化
- **业务监控**：用户采纳率、反馈评分、任务完成率

### 数据回流
- 收集真实用户数据，筛选高质量样本
- 建立 Bad Case 分析机制
- 定期重新微调，持续提升效果

### 迭代周期
```
线上观察 2 周 → 收集反馈 → 分析 Bad Case → 数据补充 → 重新微调 → 灰度验证 → 全量上线
```

## 总结

AI 全流程的核心在于：**数据质量 > 模型选择 > 训练技巧 > 部署优化**。每一步都是工程问题，需要耐心和细致的实验。
