# API 调试工具：Postman vs Insomnia vs HTTPie

API 调试是后端开发者和前端开发者的日常。无论是写接口还是调用接口，一个好用的 HTTP 客户端能让调试效率翻倍。这个领域 Postman 一家独大了很多年，但最近几年涌现出不少强力竞争者。本文评测 Postman、Insomnia、HTTPie、Bruno 和 VS Code REST Client 五种方案。

## Postman：大而不倒

Postman 是 API 调试领域的绝对王者，全球超过 2000 万开发者使用。它已经远远超出了一个「HTTP 客户端」的定义，成为一个完整的 API 平台。

Postman 的核心概念是 Collections（集合）和 Environments（环境）。Collection 把相关的 API 请求组织在一起，可以添加文件夹层级、文档描述、示例响应。Environment 管理变量（baseUrl、token、userId 等），切换环境就切换到不同的变量值组——本地开发一套变量、测试环境一套变量、生产环境一套变量，一个 Collection 走遍所有环境。

Pre-request Script 和 Test Script 是 Postman 的高级功能。Pre-request Script 在请求发送前执行 JavaScript（比如用 HMAC 签名请求体、自动刷新 Token），Test Script 在收到响应后执行（断言状态码、验证 JSON Schema、提取变量链式调用）。这本质上是把一个 API 调试工具做成了可编程的自动化测试平台。

Mock Server 根据请求示例自动生成假数据 API，前端开发可以在后端未就绪时先行开发。API Documentation 从 Collection 自动生成漂亮的文档页面。Team Workspaces 支持团队共享 Collection 和 Environment。Newman（命令行运行器）让你在 CI/CD 流水线中运行 Postman Collection 做集成测试。

问题在哪里？Postman 是 Electron 应用，内存占用大，打开慢。越来越多的功能需要登录 Postman 账号（即使个人使用也逐步引导你登录）。免费版在持续缩减（Scratch Pad 模式限制增多）。近几年的更新让老用户感觉产品越来越臃肿——它从一个小巧的 Chrome 扩展变成了一个无所不包的平台，但大部分开发者只需要其中的 20%。如果你需要团队协作、自动化测试和完整的 API 生命周期管理，Postman 依然是首选。如果你只是个人开发调试，它过重了。

## Insomnia：干净利落的替代方案

Insomnia 是 Postman 最直接的竞争对手。同样是 Electron 应用，但设计理念更干净，功能更聚焦。如果你觉得 Postman 太臃肿，Insomnia 大概率会让你舒服很多。

Collection 和 Environment 的概念与 Postman 类似，上手零学习成本。Insomnia 的差异化优势在于 GraphQL 和 gRPC 的原生支持。GraphQL 请求有 Schema 自动补全、查询构建器、片段管理。gRPC 支持 Protobuf 文件导入和交互式调用。如果你同时维护 REST、GraphQL 和 gRPC 三种 API，Insomnia 在一个应用内搞定所有。

Insomnia 的设计细节更讨喜：请求历史自动保存、响应视图多样化（Pretty/ Raw/ Preview/ Headers/ Timeline）、时间线显示请求的各个阶段耗时（DNS 解析、TCP 连接、TLS 握手、首字节时间），这对性能调优很有帮助。

Insomnia 的核心是开源的（Apache 2.0 协议），Cloud Sync 和团队功能是付费的（Insomnia Plus $5/月）。开源意味着你可以自己查看代码、构建私有版本。免费版功能已经覆盖个人开发者的绝大部分需求。

不足：插件生态不如 Postman 成熟，Mock Server 和 API 文档生成不如 Postman 自动化程度高，企业级功能（审计、RBAC）缺失。

**最适合**：REST + GraphQL + gRPC 混合使用、偏好开源、觉得 Postman 太重的开发者。

## HTTPie：终端优先的优雅选择

HTTPie 是一款命令行 HTTP 客户端，以「最人性化的 curl 替代品」著称。如果你习惯在终端工作，HTTPie 是目前体验最好的命令行 HTTP 工具。

和 curl 相比，HTTPie 的优势在于语法直观和着色输出。一个简单对比：

```bash
# curl
curl -X POST https://api.example.com/users \
  -H "Authorization: Bearer token123" \
  -H "Content-Type: application/json" \
  -d '{"name": "John", "email": "john@example.com"}'

# HTTPie
http POST api.example.com/users \
  Authorization:"Bearer token123" \
  name=John email=john@example.com
```

HTTPie 的响应输出自动格式化和着色：JSON 自动缩进和语法高亮、响应头分组显示、状态码颜色区分（2xx 绿色、3xx 蓝色、4xx 黄色、5xx 红色）。不需要像 curl 那样自己管道到 `jq` 或 `python -m json.tool`。

HTTPie 也有桌面版（Electron，$10/月），但说实话，HTTPie 的灵魂在终端。桌面版的定位类似 Postman/Insomnia 的 GUI 工具，但在功能深度上不如前两者。如果你选择 HTTPie，主要是因为命令行体验，桌面版只是一个辅助查看器。

Session 功能是 HTTPie 的独特优势：命名一个会话后，HTTPie 会保留该会话的 Cookie 和 Header，后续请求自动携带。模拟用户登录后的完整操作流程时极其实用。

**最适合**：终端重度用户、喜欢命令行优雅体验、不需要 GUI 团队协作的开发者。

## Bruno：开源 + Git 友好 + 离线优先

Bruno 是 2023 年出现的新秀，定位非常独特：Collection 以纯文本文件（Bru 格式）存储在本地目录。这意味着你可以把 API Collection 纳入 Git 版本管理，团队成员在各自本地打开，没有云同步的需求。

Bruno 完全离线工作，不需要登录，不需要账号。隐私方面做到极致——你的所有请求、环境变量、响应历史都在本地存储。开源（MIT 协议），社区活跃。

Bruno 的功能集比 Postman 精简：Environment / Variables、Pre/Post Request Scripts、Assertions、批量运行。缺少 Mock Server、API 文档自动生成、团队协作空间。但它的核心理念——Collection as Code——对重视 Git 工作流的团队来说极具吸引力。

社区相对较小，插件和集成资源不如 Postman 丰富。如果你追求稳定和生态丰富度，Postman/Insomnia 更稳；如果你追求 Git 管理和离线优先的新范式，Bruno 值得关注。

**最适合**：重视隐私和离线使用、想要 Git 管理 API 请求、开源社区参与者。

## VS Code REST Client：编辑器内的 API 调试

如果你已经在 VS Code 里写了半天代码，再切出去开 Postman 确实是一种打断。VS Code REST Client 插件让你直接在 `.http` 文件中写请求并执行：

```http
### 获取用户列表
GET https://api.example.com/users
Authorization: Bearer {{token}}
Content-Type: application/json

### 创建用户
POST https://api.example.com/users
Content-Type: application/json

{
  "name": "John",
  "email": "john@example.com"
}
```

点击请求上方的「Send Request」链接，响应在右侧面板显示。`.http` 文件可以提交到 Git，团队成员共享项目 API 调试文件。支持变量（`{{variable}}`）、环境文件（`.env`）、断言和 cURL 导入。

最大优势：零额外应用。你已经在 VS Code 里，调试 API 不离开编辑器。对前后端分离的项目，`.http` 文件可以作为「活的 API 文档」，新同事直接打开运行就能理解接口。

不足：没有 GUI 变量管理（一切靠文本），没有集合运行器（不能批量跑），没有 Mock Server 和文档生成。它是轻量工具，不做平台级的事。

**最适合**：VS Code 重度用户、喜欢简单、API 调试需求不复杂的前端/全栈开发者。

## 快速对比

| 维度 | Postman | Insomnia | HTTPie | Bruno | REST Client |
|------|---------|----------|--------|-------|-------------|
| 类型 | Electron GUI | Electron GUI | CLI (+ 可选 GUI) | Electron GUI | VS Code 插件 |
| 开源 | 否 | 核心开源 | 是 | 是 | 是 |
| 离线工作 | 受限 | 支持 | 支持 | 完全离线 | 支持 |
| GraphQL | 支持 | 原生最佳 | 不支持 | 支持 | 不支持 |
| gRPC | 有限 | 原生支持 | 不支持 | 不支持 | 不支持 |
| Git 友好 | 导出 JSON | 导出 JSON | 无集合概念 | Bru 纯文本 | .http 纯文本 |
| 价格 | 免费（有限制） | 免费 + $5/月 | CLI 免费 | 免费 | 免费 |

## 推荐

- 团队/企业 API 开发：**Postman**。不管有多少缺点，Postman 的协作、测试和文档生态是最成熟的，企业需要的就是稳定和广泛的人才库。
- 个人开发者 REST + GraphQL：**Insomnia**。比 Postman 轻，体验好，核心开源，GraphQL 支持最佳。
- 终端优先：**HTTPie**。curl 的最佳替代品，命令行体验优雅。
- 重视 Git 和离线：**Bruno**。Collection as Code 的理念适合 GitOps 工作流。
- VS Code 一切：**REST Client 插件**。不离开编辑器就是最大的效率。
