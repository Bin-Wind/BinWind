# JVM 内存模型与 GC 基础

JVM 的内存管理和垃圾回收（Garbage Collection）是 Java 程序员进阶的必修课。理解 JVM 如何分配内存、何时触发 GC、如何调优，是解决内存泄漏、OOM（OutOfMemoryError）和 Full GC 停顿问题的关键。

## JVM 内存分区

JVM 运行时数据区分为**线程私有**和**线程共享**两大类：

```
┌─────────────────────────────────────┐
│            JVM Process              │
├──────────┬──────────┬───────────────┤
│ Thread 1 │ Thread 2 │ Thread 3      │  ← 线程私有
│ ┌──────┐ │ ┌──────┐ │               │
│ │ Stack│ │ │ Stack│ │  PC Register  │
│ └──────┘ │ └──────┘ │               │
├──────────┴──────────┴───────────────┤
│              Heap                   │  ← 线程共享
│  ┌──────────┬──────────┬─────────┐ │
│  │Young Gen │ Old Gen  │ Metadata│ │
│  │(Eden+S0) │(Tenured) │ Space   │ │
│  └──────────┴──────────┴─────────┘ │
└─────────────────────────────────────┘
```

### 各区域职责

| 区域 | 归属 | 内容 | OOM 触发条件 |
|------|------|------|-------------|
| **Heap（堆）** | 线程共享 | 所有对象实例和数组 | `-Xmx` 限制被耗尽 |
| **Stack（栈）** | 线程私有 | 栈帧（局部变量表、操作数栈、方法出口） | 递归过深导致 `StackOverflowError` |
| **Metaspace（元空间）** | 直接内存 | 类的元信息、方法、常量 | `-XX:MaxMetaspaceSize` 耗尽 |
| **Program Counter** | 线程私有 | 当前线程执行的字节码行号 | 不会 OOM |

## 堆内存结构（分代模型）

堆分为**新生代**（Young Generation）和**老年代**（Old Generation）：

```
Heap
├── Young Generation (默认占堆 1/3)
│   ├── Eden 区 (默认占 Young 8/10)
│   ├── Survivor 0 (S0)  (默认占 Young 1/10)
│   └── Survivor 1 (S1)  (默认占 Young 1/10)
└── Old Generation (默认占堆 2/3)
```

对象分配与晋升流程：
1. 新对象优先分配到 Eden 区
2. Eden 区满时触发 **Minor GC（Young GC）**，存活对象复制到 Survivor 区
3. 对象在 S0 / S1 之间每经历一次 Minor GC，年龄（Age）加 1
4. 年龄达到 `-XX:MaxTenuringThreshold`（默认 15）后，晋升到 Old Generation
5. Old Generation 满时触发 **Full GC（Major GC）**，回收整个堆

## Minor GC vs Full GC

| 特征 | Minor GC | Full GC |
|------|----------|---------|
| 触发区域 | 仅 Young Generation | 整个堆 + Metaspace |
| 触发条件 | Eden 区满 | Old 区满、System.gc()、Metaspace 满 |
| 停顿时间 | 毫秒级（较快） | 秒级（较慢，STW） |
| 回收算法 | 复制（Copying） | 标记-清除 / 标记-整理 |
| 频率 | 频繁 | 应尽量避免 |

Full GC 造成的 **STW（Stop-The-World）** 会使所有应用线程暂停。调优的核心目标是**减少 Full GC 频率**和**缩短 GC 停顿时间**。

## G1GC 垃圾收集器

G1（Garbage-First）是 JDK 9+ 的默认 GC，适合 4GB 以上的大堆内存。它将堆划分为大小相等的 **Region**（默认 2048 个），每个 Region 可以是 Eden、Survivor 或 Old 角色。

```bash
# 启用 G1GC（JDK 9+ 默认）
java -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=200 \        # 期望最大停顿时间
     -XX:G1HeapRegionSize=4m \         # Region 大小
     -XX:InitiatingHeapOccupancyPercent=45 \  # 触发 Mixed GC 的堆占用阈值
     -Xms4g -Xmx4g \
     -jar app.jar
```

G1 的回收分为 **Young GC**（仅回收 Eden + Survivor Region）和 **Mixed GC**（回收部分 Old Region）。G1 的核心创新在于**可预测的停顿时间**：用户设定目标停顿时间，G1 自动选择收益最高的 Region 进行回收。

## JDK 诊断工具

JVM 内置了一套强大的诊断工具，分布在 `$JAVA_HOME/bin/` 下：

```bash
# 1. jps — 列出 Java 进程
jps -l                      # 显示进程 PID 和主类名

# 2. jstat — JVM 统计监控（最核心的 GC 监控工具）
jstat -gc <pid> 1000 10     # 每 1 秒输出 GC 数据，共 10 次
# 关注列：YGC(Young GC次数), YGCT(Young GC耗时), FGC(Full GC次数), FGCT(Full GC耗时)

jstat -gcutil <pid> 1000    # 百分比形式展示各区域使用率

# 3. jmap — 堆内存快照
jmap -heap <pid>            # 打印堆配置和各区域使用情况
jmap -dump:format=b,file=heap.hprof <pid>  # 导出 heap dump

# 4. jstack — 线程快照
jstack <pid>                # 打印所有线程状态，诊断死锁
jstack -l <pid>             # 额外输出锁信息

# 5. jcmd — 综合诊断命令
jcmd <pid> VM.flags         # 查看实际 JVM 参数
jcmd <pid> GC.heap_dump /tmp/heap.hprof  # 导出 heap dump
```

## 调优入门

调优三要素：**选择 GC 组合**、**调整堆大小**、**设定 GC 触发阈值**。

```bash
# 典型 Web 服务 JVM 参数
java -server \
     -Xms2g -Xmx2g \                          # 堆初始值和最大值设为相等，避免动态扩缩
     -Xss256k \                                # 线程栈大小
     -XX:MetaspaceSize=256m \                  # Metaspace 初始大小
     -XX:MaxMetaspaceSize=512m \               # Metaspace 上限
     -XX:+UseG1GC \                            # 使用 G1GC
     -XX:MaxGCPauseMillis=200 \                # 目标停顿 200ms
     -XX:+PrintGCDetails \                     # 打印 GC 详情
     -XX:+PrintGCDateStamps \                  # GC 日志带时间戳
     -Xloggc:/var/log/myapp/gc-%t.log \        # GC 日志输出路径
     -XX:+UseGCLogFileRotation \               # GC 日志轮转
     -XX:NumberOfGCLogFiles=5 \
     -XX:GCLogFileSize=20m \
     -XX:+HeapDumpOnOutOfMemoryError \         # OOM 时自动导出 heap dump
     -XX:HeapDumpPath=/var/log/myapp/ \
     -jar app.jar
```

> 调优原则：**不要凭空猜测 GC 问题**。先通过 jstat / GC 日志确认是否存在 FGC 频繁、停顿时间过长等问题；结合 jmap dump 分析堆内对象分布；调整参数后务必压测验证。常见的优化手段有：增大 Young Generation 比例减少对象过早晋升、提高 Mixed GC 触发阈值减少 FGC、直接用 ZGC（JDK 15+）实现亚毫秒级停顿。
