# Node.js 事件循环 (Event Loop) 深度解析

事件循环是 Node.js 的运行时核心，也是面试中几乎必问的考点。它决定了代码的执行顺序，理解它对于写出高性能、可预测的 Node.js 应用至关重要。本文将深入分析事件循环的每个阶段及微任务机制。

## Event Loop 六大阶段

Node.js 的事件循环分为六个阶段，每个阶段都有一个 FIFO 回调队列：

```
   ┌───────────────────────────┐
┌─>│           timers          │  setTimeout / setInterval 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │     pending callbacks     │  系统操作（如 TCP 错误）的延迟回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare       │  仅内部使用
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           poll            │  获取新的 I/O 事件；执行 I/O 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           check           │  setImmediate 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
└──┤      close callbacks      │  socket.on('close') 等关闭事件
   └───────────────────────────┘
```

**关键：每个阶段执行完毕后，在进入下一阶段之前，Node.js 会清空两个微任务队列。**

## 微任务 (Microtasks)

微任务在每两个阶段之间执行，有两个队列：

1. **nextTick 队列**：`process.nextTick()` 注册的回调
2. **Promise 队列**：`.then()` / `.catch()` / `async/await` 注册的回调

**nextTick 优先级高于 Promise**，在每个阶段之间，nextTick 队列会被彻底清空，然后清空 Promise 队列。

```javascript
console.log('1. 同步代码开始');

Promise.resolve().then(() => console.log('3. Promise'));
process.nextTick(() => console.log('2. nextTick'));

console.log('4. 同步代码结束');

// 输出顺序：
// 1. 同步代码开始
// 4. 同步代码结束
// 2. nextTick
// 3. Promise
```

## setTimeout(fn, 0) vs setImmediate(fn)

这两者的执行顺序取决于调用上下文。

```javascript
// 情况一：在主模块中调用（顺序不确定，受性能影响）
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));

// 情况二：在 I/O 回调中调用（setImmediate 一定先执行）
const fs = require('fs');

fs.readFile(__filename, () => {
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
  // 输出顺序：先 immediate，后 timeout
});
```

**原因：** I/O 回调在 poll 阶段执行。poll 阶段之后直接进入 check 阶段（执行 setImmediate），而 setTimeout 需要等到下一次循环的 timers 阶段。

## 微任务饥饿问题

如果微任务持续产生新的微任务，事件循环将永远卡在当前阶段之间，后续阶段永远不会执行。

```javascript
// 危险示例：无限添加 nextTick
function starve() {
  process.nextTick(() => {
    console.log('又被执行了');
    starve(); // 永远在 nextTick 队列
  });
}
starve();

setTimeout(() => {
  console.log('永远不会输出这行');
}, 0);
```

在实际开发中，避免在 nextTick 或 Promise 中递归添加自身，Node.js 内置检测到过多次 nextTick 调用会发出警告 `MaxListenersExceededWarning`。

## 实际代码详解

```javascript
const fs = require('fs');

console.log('A');

setTimeout(() => console.log('B'), 0);

setImmediate(() => console.log('C'));

fs.readFile(__filename, () => {
  console.log('D');

  setTimeout(() => console.log('E'), 0);
  setImmediate(() => console.log('F'));

  process.nextTick(() => console.log('G'));
  Promise.resolve().then(() => console.log('H'));
});

process.nextTick(() => console.log('I'));
Promise.resolve().then(() => console.log('J'));

console.log('K');

// 输出顺序：
// A, K           → 同步代码
// I, J           → 微任务（nextTick 先于 Promise）
// (B 或 C 不确定) → timers 或 check 阶段
// D              → poll 阶段（I/O 回调）
// G, H           → I/O 回调中的微任务
// F              → check 阶段（setImmediate，poll 后立即）
// E              → 下一轮 timers 阶段
```

## setImmediate 的使用场景

`setImmediate` 本质上是把一段代码推迟到当前 poll 阶段之后执行，常用于需要"让出"当前事件循环阶段的情况。

```javascript
// 场景：在递归中防止栈溢出
const list = Array.from({ length: 10000 }, (_, i) => i);

function processLargeArray(items, callback) {
  if (items.length === 0) return callback();

  const chunk = items.splice(0, 100);
  chunk.forEach(item => { /* 处理每个元素 */ });

  // 将下一批处理推迟到事件循环的下一轮，避免阻塞
  setImmediate(() => processLargeArray(items, callback));
}

processLargeArray(list, () => console.log('处理完成'));
```

## 实践建议

1. **不要阻塞事件循环。** 避免在主线程做 CPU 密集型计算（如大量循环、复杂正则回溯），可以通过 `worker_threads` 或分片 + `setImmediate` 解决。
2. **优先使用 Promise 而非 nextTick。** nextTick 更"霸道"，容易造成饥饿。
3. **理解 I/O 中 setImmediate vs setTimeout 的顺序。** 在 I/O 回调中 `setImmediate` 总是先执行。
4. **用 `setImmediate` 分片处理大量同步任务。** 每个分片之间事件循环有机会处理 I/O。

## 总结

事件循环是理解 Node.js 异步行为的地图。记住核心规则：同步代码 → 微任务（nextTick > Promise）→ 六大阶段循环。掌握这些后，你将能精确预测任何代码的执行顺序，也能避免微任务饥饿等常见陷阱。
