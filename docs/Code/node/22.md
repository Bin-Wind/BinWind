# 性能优化实战：内存泄漏排查

Node.js 应用的内存泄漏通常不像传统语言那样明显 —— 因为有 GC（垃圾回收），内存不会立即用完，但要命的是它会在生产环境慢慢增长，最终导致进程 OOM（Out of Memory）崩溃。本文带你一步步诊断和生产排查内存泄漏。

## 常见内存泄漏原因

| 原因 | 说明 | 典型表现 |
|------|------|----------|
| 全局变量 | 未声明的变量自动成为全局对象属性 | 数据永远不会被 GC |
| 遗忘的定时器 | setInterval 未 clear，闭包持有引用 | 内存持续增长 |
| 事件监听器未移除 | EventEmitter 累加 listener | 监听器数量不断增长 |
| 闭包陷阱 | 闭包引用了大对象，但只用了一小部分 | 大对象无法释放 |
| 缓存无上限 | 往 Map/Object 里不断 push 数据 | 内存线性增长 |
| 数据库连接未释放 | 连接池耗尽，每条连接都占用内存 | 连接数达到上限 |

## 代码示例：四种常见泄漏

```javascript
// 1. 全局变量泄漏
app.get('/leak1', (req, res) => {
  leakedData = { /* 大对象 */ };  // ❌ 没有 let/const，变成 global.leakedData
  res.send('ok');
});

// 2. 遗忘的定时器
const cache = {};
app.get('/leak2', (req, res) => {
  const key = req.query.key;
  cache[key] = new Array(10000).fill('*');
  // 缓存永远不清理，内存持续增长
  res.send('cached');
});

// 3. 事件监听器未移除
const EventEmitter = require('events');
const emitter = new EventEmitter();

app.get('/leak3', (req, res) => {
  emitter.on('data', (data) => {
    // ❌ 每次请求添加新的监听器，从不移除
    console.log(data);
  });
  res.send('ok');
});

// 4. 闭包持有大对象
function createHandler() {
  const bigData = Buffer.alloc(100 * 1024 * 1024); // 100MB
  return function(req, res) {
    // ❌ 只用到了 req.url，但闭包让 bigData 无法被 GC
    res.send(`URL: ${req.url}`);
  };
}
app.use(createHandler());
```

## 修复方案

```javascript
// 1. 全局变量 → 使用 let/const + 严格模式
'use strict';
app.get('/fix1', (req, res) => {
  const data = { /* 大对象 */ };  // 请求结束时 data 可被 GC
  res.send('ok');
});

// 2. 缓存 → 使用 LRU Cache + 上限
const LRU = require('lru-cache');
const cache = new LRU({ max: 500 });
app.get('/fix2', (req, res) => {
  cache.set(req.query.key, computeValue());
  res.send('ok');
});

// 3. 事件监听 → 使用 once() 或手动 removeListener
app.get('/fix3', (req, res) => {
  emitter.once('data', (data) => {  // once 自动移除
    console.log(data);
  });
  // 或手动清理
  const handler = (data) => console.log(data);
  emitter.on('data', handler);
  emitter.removeListener('data', handler);
  res.send('ok');
});

// 设置最大监听器数警告（防止泄漏前兆）
emitter.setMaxListeners(20);

// 4. 闭包 → 只保存需要的引用
function createHandler() {
  return function(req, res) {
    // 不持有大对象引用
    res.send(`URL: ${req.url}`);
  };
}
```

## Chrome DevTools 诊断

### 步骤 1：启动应用（带 inspector）

```bash
node --inspect app.js
```

### 步骤 2：打开 Chrome DevTools

访问 `chrome://inspect`，点击 "Open dedicated DevTools for Node"

### 步骤 3：Memory 标签

1. 进入 Memory 标签页
2. 选择 "Heap snapshot"
3. 在应用启动时拍一张快照（Snapshot 1）
4. 模拟流量（执行可能泄漏的操作），拍第二张快照（Snapshot 2）
5. 再次模拟流量，拍第三张快照（Snapshot 3）
6. 对比 Snapshot 1 和 Snapshot 3，查看 "Delta" 列

关注 Delta 中持续增长的对象 —— 这就是泄漏源。

### 对比快照技巧

在 Snapshot 3 的视图中，切换到 "Comparison" 模式，选择与 Snapshot 1 对比，按 "Size Delta" 降序排列。查看 Constructor 列：如果 `(closure)` 或 `(array)` 等持续增长，点击展开查看 Retainers 追溯到具体的代码。

## process.memoryUsage() 监控

```javascript
// 定时输出内存使用情况
setInterval(() => {
  const mem = process.memoryUsage();
  const fmt = (bytes) => `${(bytes / 1024 / 1024).toFixed(2)} MB`;
  console.log({
    rss: fmt(mem.rss),           // 常驻内存（总占用）
    heapTotal: fmt(mem.heapTotal), // V8 堆总量
    heapUsed: fmt(mem.heapUsed), // V8 堆已用
    external: fmt(mem.external), // C++ 对象绑定的 JS 内存
    arrayBuffers: fmt(mem.arrayBuffers || 0),
  });
}, 5000);
```

**正常模式：** `heapUsed` 锯齿状上升下降（GC 回收后回落）
**泄漏模式：** `heapUsed` 只升不降，或者每次 GC 后残留值越来越大

## 生产环境诊断：Clinic.js

Clinic.js 是 Node.js 官方推荐的性能诊断工具套件。

```bash
# 安装
npm install -g clinic

# 诊断 CPU
clinic doctor -- node app.js

# 诊断事件循环延迟
clinic bubbleprof -- node app.js

# 诊断内存泄漏
clinic heapprofiler -- node app.js
```

Clinic 会生成一个交互式 HTML 报告，火焰图中可以看到火焰持续扩大的区域。

## 完整诊断流程示例

```bash
# 1. 确认进程内存正在增长
ps aux | grep node

# 2. 添加内存监控端点
cat >> app.js << 'EOF'
app.get('/_memory', (req, res) => {
  const mem = process.memoryUsage();
  res.json({
    heapUsed: (mem.heapUsed / 1024 / 1024).toFixed(2) + 'MB',
    rss: (mem.rss / 1024 / 1024).toFixed(2) + 'MB',
  });
});
EOF

# 3. 用 ab 或 wrk 压测
wrk -t4 -c100 -d30s http://localhost:3000/api/test

# 4. 观察内存端点返回值
watch -n 1 'curl -s http://localhost:3000/_memory'

# 5. 如果 heapUsed 持续上升，用 clinic heapprofiler
clinic heapprofiler -- node app.js

# 6. 在另一个终端压测，clinic 采集数据
# 按 Ctrl+C 停止，分析生成的 .html 报告
```

## 预防原则

1. **`'use strict'`**：避免意外全局变量
2. **定时器成对清理**：`setInterval` 必有 `clearInterval`
3. **事件监听用 `once`**：一次性事件不要用 `on`
4. **缓存设上限**：LRU Cache，超时淘汰
5. **监控内存趋势**：在监控平台（如 Grafana）中展示 `heapUsed` 曲线
6. **定期压测**：CI 中加入内存泄漏检测步骤

## 总结

Node.js 内存泄漏的核心诊断方法是：Heap Snapshot 对比 + 内存趋势监控。DevTools 用于本地排查，Clinic.js 用于生产采集。修复顺序：先确认泄漏（heapUsed 单边上涨）→ 定位（对比快照或火焰图）→ 修复（按常见原因逐个排查）→ 验证（再次压测确认内存回落）。养成编码阶段就预防的好习惯，比事后修补成本低得多。
