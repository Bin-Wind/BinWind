# 虚拟环境与依赖管理

Python 项目开发中，依赖管理是最容易被忽视却至关重要的一环。不同项目可能依赖同一库的不同版本，直接全局安装会引发版本冲突。虚拟环境（Virtual Environment）正是为了解决这个问题而生的隔离机制。

## 为什么需要虚拟环境

假设项目 A 需要 Django 3.2，项目 B 需要 Django 4.0，同时安装在系统 Python 中会导致冲突。虚拟环境为每个项目创建独立的 Python 解释器环境，包括独立的 `site-packages` 目录，项目之间的依赖互不干扰。

## venv：Python 内置的虚拟环境

从 Python 3.3 开始，标准库内置了 `venv` 模块，无需额外安装。

```bash
# 创建虚拟环境
python -m venv .venv

# 激活虚拟环境（macOS / Linux）
source .venv/bin/activate

# 激活虚拟环境（Windows）
.venv\Scripts\activate

# 退出虚拟环境
deactivate
```

激活后，终端提示符前会出现 `(.venv)` 标识，此时 `pip install` 安装的包将只作用于当前环境。

## pip freeze 与 requirements.txt

最传统的依赖管理方式：将已安装的包及其版本冻结到文本文件。

```bash
# 导出当前环境的所有包
pip freeze > requirements.txt
```

生成的 `requirements.txt` 内容示例：

```
Django==4.2.0
requests==2.31.0
numpy==1.26.0
```

其他开发者可以通过以下命令复现环境：

```bash
pip install -r requirements.txt
```

这种方式的缺点是：无法区分直接依赖和间接依赖，也没有开发依赖和正式依赖的分组能力。

## pyproject.toml：现代化的项目配置

PEP 518 / PEP 621 确立了 `pyproject.toml` 作为 Python 项目的标准配置文件，统一管理项目元数据、构建系统和依赖信息。

```toml
[build-system]
requires = ["setuptools>=68.0", "wheel"]
build-backend = "setuptools.build_meta"

[project]
name = "my-project"
version = "0.1.0"
description = "A sample project"
requires-python = ">=3.10"
dependencies = [
    "fastapi>=0.100.0",
    "uvicorn[standard]>=0.23.0",
]

[project.optional-dependencies]
dev = [
    "pytest>=7.0",
    "black>=23.0",
    "ruff>=0.1.0",
]
test = [
    "pytest-cov>=4.0",
    "httpx>=0.24.0",
]
```

## Poetry：现代化的依赖管理工具

Poetry 是目前最流行的 Python 依赖管理器之一，提供了类似 npm / cargo 的开发体验。

```bash
# 安装 Poetry
pip install poetry

# 初始化项目
poetry new my-project
cd my-project

# 添加依赖
poetry add requests

# 添加开发依赖
poetry add --group dev pytest black

# 安装所有依赖（根据 poetry.lock 锁定版本）
poetry install

# 更新依赖
poetry update

# 导出 requirements.txt（兼容传统部署）
poetry export -f requirements.txt --output requirements.txt
```

Poetry 使用 `poetry.lock` 锁定所有依赖树的精确版本，确保不同机器上安装的依赖完全一致。

## pip-tools：极简主义的替代方案

如果你偏好 `requirements.txt` 的工作流但想要锁定功能，`pip-tools` 是个好选择。

```bash
pip install pip-tools
```

创建 `requirements.in` 仅列出直接依赖：

```
# requirements.in
fastapi
uvicorn
sqlalchemy
```

然后编译生成锁定的 `requirements.txt`：

```bash
# 编译锁定文件
pip-compile requirements.in

# 同步环境到锁定文件
pip-sync requirements.txt
```

## 依赖管理的工程实践

1. **始终使用虚拟环境** — 每个项目一个独立环境，推荐命名为 `.venv` 并加入 `.gitignore`。
2. **区分直接与间接依赖** — `requirements.txt` 适合简单脚本，`pyproject.toml` + Poetry 适合正式项目。
3. **锁定依赖版本** — 使用 lock 文件确保团队环境一致；CI/CD 中运行 `pip install -r requirements.txt`（锁定版）而非 `pip install .`。
4. **定期更新并测试** — 依赖过时可能包含安全漏洞，使用 `pip list --outdated` 或 `poetry show --outdated` 检查。
5. **使用 .gitignore 排除虚拟环境** — 不要把 `.venv` 目录提交到版本控制。

选择合适的工具组合取决于项目规模：小脚本用 `venv + requirements.txt` 即可，中大型项目推荐 `Poetry + pyproject.toml`，基础设施工具场景下 `pip-tools` 也是一个轻量且成熟的选择。
