# 微交互 (Micro-interaction) 设计指南

微交互是用户界面中被精心设计的"微小瞬间"——按下按钮时的形态变化、下拉刷新时的加载动画、表单验证失败时的红色抖动、收到新消息时的角标弹出。它们如此微小，以至于用户在无意识中体验着它们，但如果将它们移除，整个界面会立刻变得生硬、不可爱、甚至让人不确定操作是否执行了。微交互的本质是用最轻量级的反馈，让用户感知到系统状态的改变。

## 微交互的四层结构

设计师 Dan Saffer 在《微交互》一书中提出了一个经典的模型，每个微交互由四个部分组成：

### Trigger（触发器）

启动微交互的条件。可以是用户手动触发（点击按钮、悬停在元素上、输入文字），也可以是系统自动触发（收到推送通知、下载完成、计时器到点）。

### Rules（规则）

定义了触发器被激活后会发生什么。规则是微交互的逻辑核心——例如"点击菜单按钮后，菜单从左侧滑入，同时按钮变为关闭图标"。

### Feedback（反馈）

用户感知到微交互发生的方式，这是设计师可控的视觉/听觉/触觉呈现。反馈可以是视觉的（颜色变化、大小变化、动画）、听觉的（提示音、操作音）或触觉的（手机振动）。

### Loops & Modes（循环与模式）

决定了微交互的持续时间、重复方式和在不同场景中的变化。例如，一个加载动画可能持续循环直到数据返回（无限循环），而一个成功打勾动画执行一次后停止（单次循环）。Mode 决定微交互在不同条件下的行为差异。

## 常见的微交互类型

### Hover 状态（悬停）

鼠标悬停在可交互元素上时，元素的背景色微调、阴影抬起、光标变化为 Pointer。Hover 态的持续时间通常 100-200ms，动画要轻快。在移动端没有 Hover 概念，但可以使用 While Pressing（按压时）实现类似效果。

### Active 状态（按下）

点击的瞬间——按钮微微缩放（缩小到 95-97%）、颜色加深、阴影内收。这个反馈极其重要，它让用户"感觉到"自己确实按下了按钮。

### 加载动画

当操作需要等待时（网络请求、文件处理），加载指示器告知用户系统正在工作。三种常见形式：骨架屏（Skeleton Screen，在内容加载前显示灰色占位块，暗示即将出现的内容结构）、Spinner（旋转动画，适合短等待）、进度条（显示实际百分比，适合耗时 5 秒以上的操作）。

### 成功/错误反馈

操作完成后的一瞬间：成功动画（打勾、弹出、绿色高亮）或错误提示（红色抖动、标红显示、错误信息弹出）。错误动画的设计要点是——不要惩罚用户，而是帮助用户纠正错误。抖动的幅度不宜过大（2-4px 的左右位移即可），持续时间不宜过长（200-300ms）。

### 通知和提醒

一个新消息到达时，相关图标上的数字角标会出现微小的弹跳动画，吸引注意力却不过分打断当前任务。

## 关键时间参数

微交互的节奏感完全由持续时间和缓动曲线控制：

- **持续时间**：100-500ms 是微交互的合适范围，其中 200-300ms 是"甜点区间"——够快以致不阻塞用户操作，够慢以致可被感知。超过 500ms 的动画会让用户感到迟钝。
- **更快场景（100-150ms）**：Hover 切换、Checkbox 勾选、开关切换。
- **标准场景（200-300ms）**：按钮按下、菜单展开收起、弹窗进出。
- **较慢场景（400-500ms）**：页面转场、复杂元素的出场动画、品牌展示动画。

## 缓动曲线

缓动（Easing）决定了动画在时间进程中的速度变化，它对微交互的"感觉"有决定性的影响：

- **Ease-out（快进慢出）**：动画开始时速度快，结束时速度降低。适合元素进入场景（淡入、滑入），因为我们希望内容尽快出现，最后的减速提供一个温和的结束感。
- **Ease-in（慢进快出）**：开始时慢，结束时快。适合元素离开场景（淡出、滑出），因为我们不想在离开上花太多时间。
- **Ease-in-out（慢进慢出）**：两端慢、中间快。适合不涉及"进入/离开"语义的中性运动，如旋转、颜色过渡。
- **Spring（弹性动画）**：在结束前进行小幅度的超越和回弹，带来更有弹性的质感。适合品牌动画、趣味性 UI。
- **Linear（匀速）**：不推荐使用。自然界中几乎没有匀速运动，人眼会立刻感知到"机械感"和"不对头"。

## 微交互的目的（而非装饰）

微交互不是"让界面动起来好看"的装饰。每一个微交互都应当服务于至少一个明确的目的：

- **提供反馈**——让用户知道操作已被系统接收并处理。
- **展示系统状态**——加载中、已连接、离线、出错。
- **引导注意力**——将用户视线引向新出现或发生变化的内容（减少用户的视觉搜索成本）。
- **创造愉悦**——有时，一个精巧的微交互可以让用户在使用产品的过程中获得意外的满足感（如 Twitter 心形点赞的粒子爆炸效果）。

好的微交互设计是隐形的——用户不需要意识到它的存在，但移除它的那一刻，用户会明显感觉"这个产品好像哪里不对"。

## Do's and Don'ts

**Do（要做）：**
- 保持微交互的意图明确：一个操作对应一个明确的反馈动画。
- 确保所有动画不会阻塞用户操作：动画运行期间用户应仍能进行下一步交互。
- 考虑 prefers-reduced-motion：某些用户因感官敏感性或晕动症需要减少动画，通过系统设置可以关闭。
- 使用 Figma 原型模式和开发规范一致的语言来描述动画参数（duration、easing、delay）。
- 在组件文档中记录微交互规范：触发条件、动画时长、缓动曲线。

**Don'ts（不要做）：**
- 不要过度动画化：每个页面元素都有一个"弹出"动画会让用户感到疲惫和不专业。
- 不要让加载动画成为用户等待的唯一反馈——如果等待不可避免，给出时间估算。
- 不要把动画时长设置得超过 500ms，除非是品牌级展示或沉浸式体验。
- 不要只设计"理想状态"的微交互——网络慢、操作失败等异常态的微交互同样需要设计。

## 交付工具

微交互在设计稿中只能走这么远——Figma 的原型功能可以模拟简单的过渡和动画，但对于逐帧动画、Lottie（After Effects 导出的 JSON 动画）等复杂微交互，设计师需要借助专用工具交付给开发：

- **Lottie / LottieFiles**：After Effects 中制作动画，通过 Bodymovin 插件导出为 Lottie JSON 文件，开发使用 lottie-web / Lottie-iOS / Lottie-Android 库直接渲染。适合复杂的图标动画、加载动画、品牌动画。
- **Rive**：新兴的交互式动画工具，设计师可以在 Rive 编辑器中使用状态机直接定义交互逻辑，导出文件轻量且开发集成简单。适合需要响应输入的动态图形。
- **Principle / ProtoPie**：高保真交互原型工具，生成的交互演示可以导出视频作为开发参考，但不直接输出代码资源。

设计微交互时的核心交付物是**动效规格说明**（Animation Spec），内容包括：触发事件、受影响元素、属性变化（opacity、scale、position、rotation 等）、持续时间、缓动曲线、延迟时间。清晰的动效说明让开发实现和设计预期对齐，减少返工。

## 经典微交互实例

- **Twitter 点赞心形**：点击时心形放大 120%，填充红色，同时向外爆发一圈彩色粒子，随后回弹至标准尺寸。Duration 约 300ms，使用 Spring 缓动。
- **下拉刷新**：用户下拉列表时，顶部的刷新控件从"下拉刷新"变为"释放刷新"，松手后变为旋转加载动画，数据返回后动画消失、列表弹回。这个微交互包含了 Trigger（下拉）、Rules（超过阈值触发刷新）、Feedback（加载动画+列表弹 回）、Loop（持续加载直到完成）。
- **密码强度指示器**：在注册表单输入密码时，下方的强度条根据密码复杂度从红色（弱）到橙色（中）到绿色（强）变化，伴随宽度扩展动画。这是"即时反馈"的经典案例——让用户在输入完成前就知道信息是否合格。
