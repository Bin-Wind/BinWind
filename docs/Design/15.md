# 响应式设计：多端适配策略

今天的用户可能在手机、平板、笔记本、甚至智能电视上访问同一个产品。响应式设计不是"让网站在手机上也能看"，而是为每一种设备提供最佳体验。

## Breakpoint 策略

常见的断点区间划分：Mobile 为 320px 到 767px（从小屏手机到较大手机），Tablet 为 768px 到 1023px（iPad 竖屏到横屏、小尺寸笔记本），Desktop 为 1024px 及以上（笔记本、显示器）。但这些不是固定规则——断点应该由内容驱动。当布局"撑不住"了再加断点，而不是套用一套预设的设备尺寸。

常见补充断点包括 480px（大屏手机横屏）和 1440px（大桌面屏幕）。关键原则：看着你的内容，拉伸浏览器窗口，在内容开始变形的宽度加断点——而不是说"手机是 375px，我就设 375px 的断点"。不同项目的断点位置应该不同，因为不同内容在不同宽度下的表现各异。一个以数据表格为主的 B 端产品和以图文内容为主的 C 端产品，它们的断点不可能相同。

## Fluid vs Adaptive 设计

两种方法各有优劣。**流式 (Fluid)** 使用百分比和 vw/vh 单位构建布局，元素随视口等比缩放。优点是平滑无间隙，在任何宽度下都能填充屏幕。缺点是在极端宽度下控制力较弱——超宽屏上内容可能过于稀疏、行长达几百字难以阅读，极窄屏上内容可能挤成一团。

**自适应 (Adaptive)** 使用固定的断点，在每个断点处触发离散的布局变化。优点是每个设备段都有精确控制，设计师可以为每个断点精细调整。缺点是断点之间存在间隙——比如 767px 和 768px 之间虽然只差 1px，但布局完全不同，这种硬切分在特定窗口尺寸下体验不佳。

现代最佳实践是混合方案：在自适应布局容器内使用流式排版和图片。用 max-width 容器（常见 1200px、1440px）防止超宽显示器上的布局无限制拉伸——全屏 3000px 宽的 16 字一行文字没人读得舒服。文字用 clamp() 实现流式缩放，但区块布局在关键断点处做结构性调整。这就是"断点做架构，流体做细节"的思路。

## Mobile-First 设计流程

为什么 Mobile-First 行之有效：首先，它强制做优先级排序——空间有限，你必须决定什么才是真正重要的，这对所有屏幕尺寸都是好事。如果你能在 375px 宽的空间里排好优先级，放到 1440px 你只会做得更好。其次，性能天然受益——移动端的约束促使代码更精简、资源更克制，这些优化同样惠及桌面端用户。第三，渐进增强：在移动端提供核心体验，随屏幕增大逐步添加功能和复杂度，从"人人都能用"到"大屏更好用"。

流程上，先设计移动端布局，然后用 min-width 媒体查询依次为更大屏幕添加复杂度。永远不要用 max-width 来"降级"桌面设计——这种从大到小的思路是本末倒置。CSS 示例：基础样式按移动端写，然后 `@media (min-width: 768px) { ... }` 添加平板布局，`@media (min-width: 1024px) { ... }` 添加桌面布局。这种方式意味着旧浏览器或不支持媒体查询的设备至少能得到可用的移动端体验。

## 布局变化的节奏

典型的布局递进：320-767px 采用单列、垂直堆叠、汉堡菜单（导航隐藏以节省空间）。768-1023px 采用双列布局，水平导航开始出现，侧边栏从底部或顶部移到侧边。1024-1279px 采用三列加侧边栏，全局导航完整可见，留白变得充裕。1280px 及以上使用多列配合 max-width 容器，有意识地运用留白来区分内容区。需要强调的是：列数应该由内容的需要来证明，而不是"因为这里有空间所以加一列"。不必要的多列只会增加认知负担——用户需要决定按什么顺序阅读各列，这本身就是额外的脑力消耗。

## 响应式排版

使用 CSS clamp() 实现流式排版。正文字号：`font-size: clamp(1rem, 0.8rem + 0.5vw, 1.25rem)`——最小 16px 保证可读性，随视口平滑放大，最大 20px 防止在超大屏上文字过大。标题使用带有流式值的字号层级，h1 在桌面端要够大以建立清晰的视觉层级，在移动端要适当收敛以免一个字就换行。

行高：正文 1.5（移动端可接受 1.4），标题 1.2-1.3。行高也需要响应式——移动端小屏幕上行高太大显得松散，桌面端宽屏幕上行高太小阅读困难。特别注意：移动端正文字号不得小于 16px——iOS Safari 在表单输入框获得焦点时，如果页面的 font-size 小于 16px，会自动放大页面，破坏布局体验。这不是 bug 而是 Safari 的有意行为，适应它而不是对抗它。

## Touch Target 标准

移动端可用性的硬指标：Apple HIG 要求最小 44x44pt 触控区域，Material Design 要求最小 48x48dp。这些数字来自人体工学——成年人的食指指尖宽度约 10-12mm，对应约 40-48px。鼠标驱动的界面可以接受 24-28px 的交互元素，因为鼠标精度远高于手指。

触控目标之间的间距至少 8px，以降低误触率。实践中这意味着：表单按钮高度至少 44px，并在视觉边界之外留有可点击的 padding；导航项垂直间距足够防止手指同时触碰两个目标；列表操作（如删除、编辑）的按钮足够大，确保用户不会点错。看起来再精致的按钮，如果手指点不到就没有意义。设计评审时用实际触控测试——在手机上打开演示，看能不能一次性准确地完成关键操作。

## 响应式图片

三种核心技术。其一，**srcset + sizes**：让浏览器根据视口宽度和像素密度从一组图片中选择最合适的一张。写法如 `srcset="img-320.jpg 320w, img-640.jpg 640w, img-1280.jpg 1280w"` 配合 `sizes="(max-width: 640px) 100vw, 50vw"`。浏览器会考虑设备像素比和网络状况，自动选择最优尺寸。

其二，**`<picture>` 元素用于场景裁剪 (art direction)**：在不同断点需要不同画面构图（不只是不同大小）时使用——桌面用横版构图展示全景，手机用正方形或竖版构图聚焦主体。不能简单地缩放同一张图片来适配所有屏幕，构图会完全改变信息传达。

其三，**懒加载**：折叠线以下的图片使用 `loading="lazy"` 属性，首屏大图使用 `loading="eager"`。配合 `decoding="async"` 可以进一步优化主线程性能。格式方面，照片类用 WebP（配合 JPEG/PNG 降级方案），图标和插画类用 SVG。使用 `<picture>` 结合 `type="image/webp"` 提供格式降级。

## 测试方法

浏览器 DevTools 的设备工具栏用来做快速检查，但请注意这些只是模拟——触控行为、渲染差异和实际性能在真实设备上完全不同。真机测试是不可替代的，至少要在一台 iOS 和一台 Android 设备上验证，涵盖中端机和高端机。BrowserStack 和 LambdaTest 提供基于云的真机测试，适合需要覆盖大量设备型号的场景。

别忘了网络限速测试——在 3G 条件下加载页面，理解移动用户在电梯里、地铁里的真实体验。测试清单：所有触控目标是否大于 44px？文字是否在所有宽度下都可读？表单输入是否方便（不需要缩放和水平滚动）？页面是否在任何宽度下都不出现水平滚动条？图片是否在合适的断点加载了合适的尺寸？

## Figma 中的响应式设计

Figma 的 Constraints 允许将元素固定在父级边缘，使其在容器尺寸变化时按预期缩放。Auto Layout 可以构建 wrap、fill 或 hug 行为的组件，基本对应 CSS 的 flexbox 行为。在 Auto Layout Frame 上设置最小/最大宽度可以模拟断点行为。

实践建议：为每个主要断点（移动端、平板、桌面端）创建独立的 Frame，以清晰传达布局意图给开发者。不要仅靠 Auto Layout 来传达响应式行为——不同断点下的布局结构差异需要分别呈现，单一 Frame 无论如何拉伸都无法表达"平板是两列，手机是一列"这样的结构性变化。每个 Frame 旁边标注对应的 min-width 值，让开发一眼看懂在什么宽度触发这个布局。
