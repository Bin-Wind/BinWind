# 无障碍设计 (Accessibility) 入门

无障碍设计常常被误解为"为少数人做额外的工作"，但实际上，a11y 的受益者远比你想象的多。从手臂骨折的同事到阳光下看手机的用户，无障碍特性改善的是所有人的体验。

## 为什么 a11y 重要

三个有说服力的理由。**用户覆盖**：全球约 15% 的人口（超过 10 亿人）有某种形式的残疾，包括视力障碍（色盲、低视力、失明）、听力障碍、运动障碍和认知障碍。这不是边缘场景，而是你用户群中不可忽视的组成部分。即使暂时不考虑永久性残疾，任何人在某些情况下都会遇到"情境性残疾"——强光下看不清屏幕、一手抱孩子只能单手操作、嘈杂环境中听不到音频提示。无障碍设计就是在承认：每个人都有需要辅助的时候。

**法律合规**：各国法规日益严格——美国 ADA 诉讼数量逐年增加，2023 年美国联邦法院受理的 ADA Title III 诉讼超过 8000 起。欧盟 EAA（European Accessibility Act）于 2025 年 6 月正式生效，覆盖所有在欧盟市场销售的产品和服务。Web 无障碍不再是锦上添花，而是法律义务和进入市场的准入门槛。

**更好的 UX**：清晰的对比度、良好的键盘导航、有意义的标签——这些无障碍实践本身就改善了所有人的体验。这就是 curb-cut effect：为轮椅设计的斜坡同时也方便了推婴儿车和拉行李箱的人。无障碍不是特殊对待，而是通用设计的自然延伸。提升 a11y 的过程往往也是整顿设计债务、统一交互标准的过程。

## WCAG 标准与等级

Web Content Accessibility Guidelines (WCAG) 2.2 定义了三个合规等级。**级别 A** 是最基础的要求，不满足则产品对部分用户基本不可用。例如：图片必须有 alt 文本、不可仅靠颜色传达信息、视频必须有字幕。**级别 AA** 是业界标准目标，绝大多数组织的合规目标即为此级别。例如：文本对比度 4.5:1、键盘可访问所有功能、一致导航、表单错误有明确提示。**级别 AAA** 是最高标准，难以全面实现，通常针对特定场景。例如：对比度 7:1、手语翻译、阅读水平不超过初中程度。

实践建议：以 AA 为目标覆盖所有功能，选取 AAA 中的若干项（如更高的对比度、更详细的辅助说明）作为关键页面的增强。不要试图全面实现 AAA——这几乎是不可能的，反而会消耗大量资源却收效甚微。

## 色彩对比度

这是最常见的 a11y 失败点，也是最容易修复的。**正文文本**：最低对比度 4.5:1（AA），7:1（AAA）。**大号文本**（18px+ 常规或 14px+ 粗体）：最低对比度 3:1（AA），4.5:1（AAA）。因为大号文本的笔画更宽，在较低对比度下仍可辨识。**UI 组件与图形**：图标、输入框边框等至少需要 3:1 的对比度才能被辨识——低于这个值的控件边界会"消失"。

工具推荐：Stark（Figma/Sketch 插件，设计时即检查，避免到开发阶段才发现问题）、WebAIM Contrast Checker（免费的网页工具，能实时看到对比度数字和 AA/AAA 通过情况）、Chrome DevTools 的 CSS Overview 面板（能扫描整个页面找出所有对比度问题）。常见陷阱：灰色文字在白色背景上很容易不达标——`#999` 在 `#FFF` 上的对比度只有 2.85:1，远低于 4.5:1 的要求。很多设计师觉得"这个灰色挺清楚的"，但数字不会说谎。

还有一个关键原则：不要仅靠颜色传达信息。错误状态不能只用红色边框提示，必须同时有图标和文字说明。约 8% 的男性有色觉障碍，仅靠红绿色区分成功和失败对他们来说完全不可见。同理，图表中不同数据系列要同时用不同颜色和不同纹理或标记来区分。

## 键盘导航

并非所有用户都能使用鼠标。运动障碍用户（帕金森、关节炎、截肢等）依赖键盘或开关设备导航，盲人用户则完全依赖键盘配合屏幕阅读器。

**Tab 顺序**：确保 Tab 键按合乎逻辑的顺序在可交互元素之间移动——通常遵循视觉阅读顺序（从左到右、从上到下）。不要在视觉上从左到右的布局中出现 Tab 跳到右边再跳回左边的情况。**焦点指示器**：必须有清晰可见的 focus outline。这是许多设计师最常见的失误——为了"好看"而去掉 `outline: none` 却不提供替代样式。可以自定义焦点样式（如蓝色光环、虚线边框），但绝对不能将其设为不可见。推荐使用 `:focus-visible` 而非 `:focus`：只在键盘导航时显示焦点环，鼠标点击时隐藏，既保证了无障碍体验又不影响视觉美观。

**跳过链接 (Skip Links)**：在每个页面顶部提供"跳转到主内容"的链接，对屏幕阅读器和键盘用户至关重要——没有它，每次加载页面都要 Tab 通过一整个导航栏才能进入正文，在导航项多的站点上这意味着几十次键盘敲击。**快捷键**：提供常用功能的键盘快捷键，但要特别注意不要与屏幕阅读器或浏览器的原生快捷键冲突。优先使用复合键（如 Ctrl+Shift+K）而非单键。

## 屏幕阅读器与语义化

让内容对视障用户可访问。**语义化 HTML 优先**：使用 `<header>`、`<nav>`、`<main>`、`<article>`、`<aside>`、`<footer>` 而不是无意义的 `<div>`。每个页面应该只有一个 `<main>` 标签和一个 `<h1>`。语义化标签让屏幕阅读器能够识别页面结构，帮助用户快速在区域之间跳转。

**Heading 层级**：h1 → h2 → h3，不要跳级。屏幕阅读器用户通过标题来浏览页面结构——按 H 键在不同标题间跳跃，按 1-6 键跳转到对应级别的标题——就像目视用户通过视觉层次来扫描页面一样。跳级（如 h2 后直接用 h4）会破坏这个导航模式。

**alt 文本**：所有信息性图片必须有有意义的 alt 描述。装饰性图片使用空 alt（`alt=""`），屏幕阅读器会完全跳过它。关键点：描述图片传达的信息而非外观——"销售额在 2025 年增长 34%"而非"蓝色柱状图"。如果是复杂图表，需要同时在正文或附近提供等价的文字描述。

**ARIA 标签**：当原生 HTML 无法满足需求时使用。`aria-label` 为没有可见标签的元素提供名称，`aria-expanded` 表示可展开组件的状态。但第一条规则是：**尽可能不用 ARIA**。原生 HTML 总是比 ARIA 更可靠、更少出错。ARIA 是不完美的补丁，不是首选方案。

## 表单的无障碍设计

表单是交互的核心，也常常是页面上最不易访问的部分。**标签始终可见**：使用 `<label>` 元素关联到对应 `<input>`，点击标签也能聚焦到输入框——这扩大了触控区域，对所有用户都是好事。不要在输入框内使用 placeholder 代替 label——placeholder 在用户输入时消失，且默认的浅灰色对比度通常不达标。

**错误信息的关联**：错误提示必须与对应字段关联（使用 `aria-describedby`），这样屏幕阅读器在用户聚焦到出错字段时才能正确读出错误信息。**清晰的指令**：输入格式要求（如日期格式、密码规则、手机号格式）要提前说明而不是在用户出错后才告知——预先告知是帮助，事后告知是惩罚。**错误恢复**：提供明确方法来纠正错误，不要清空用户已填写的字段——清空已填好的内容是对用户时间的极大不尊重。**成功反馈**：表单提交后明确告知成功或失败，焦点应自动移到确认消息或第一个错误上。

## 动态与动画的可访问性

`prefers-reduced-motion` 媒体查询用于尊重系统的减少动画设置。在此设置下应禁用视差滚动、自动播放动画和大范围移动动效。不要自动播放带声音的视频，始终提供暂停和停止控制。闪烁内容避免每秒超过 3 次的闪烁——此类闪烁可能引发光敏性癫痫发作（WCAG 准则 2.3.1）。给用户控制权：动画应有暂停选项，轮播图应可手动控制，自动轮播应在用户 hover 时暂停。

## 测试方法

无障碍不是打勾项，它需要实际测试。**键盘走查**：仅用 Tab 键完成所有主要操作流程——能否看到焦点？能否完成所有操作？焦点是否被卡在某个元素中出不来（焦点陷阱）？**屏幕阅读器测试**：macOS 自带 VoiceOver（Cmd+F5），Windows 自带 NVDA（免费下载的强大屏幕阅读器）。至少学习一个屏幕阅读器的基本操作，用它走查你的核心流程——这可能是你做过的最能打开眼界的测试。

**自动化工具**：axe DevTools（浏览器插件）和 Lighthouse（Chrome DevTools）可自动检测 30-40% 的问题，包括对比度不足、缺少 label、heading 跳级等。但自动化只能发现程序化可检测的问题——它无法判断 alt 文本是否真的有意义，无法判断 Tab 顺序是否合理，无法判断交互体验是否流畅。人工测试不可或缺。**对比度检查**：使用 Stark 或 WebAIM 逐个检查所有文字与背景的颜色组合，不要漏掉任何状态。**缩放测试**：浏览器放大 200% 后页面是否仍然可用？内容不应被截断或重叠，也不应出现水平滚动条。

## 将 a11y 融入设计流程

无障碍不是最终 QA 的一个步骤，而是必须融入全流程的工作。设计阶段：在 Figma 中使用 Stark 插件检查对比度，使用 A11y - Focus Order 插件规划焦点顺序，在设计评审中把"这个的键盘行为是什么？"作为常规问题。开发阶段：使用语义化 HTML、ARIA 属性（仅在需要时）、键盘事件处理。QA 阶段：键盘走查、屏幕阅读器测试、自动化扫描三者结合。文档化：在组件文档中记录 a11y 属性和键盘行为，让所有团队成员了解每个组件的无障碍要求和预期行为。无障碍不是一次性项目，而是持续的实践——每次新增功能、每次设计迭代都应该把无障碍纳入考量，就像每次代码提交都应该通过 lint 检查一样自然。
