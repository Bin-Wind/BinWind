# 设计到开发：交付标注与切图规范

设计稿做得再好，如果开发还原度只有 70%，最终产品就是打了折的设计。高效的交付标注和规范的切图流程，是设计师和开发者之间的桥梁。本章将从交付工具、标注规范、切图流程到协作沟通，系统性地覆盖设计交付的完整工作流。

## 设计交付工具的演变

过去，设计师在 Photoshop 中手动拉红色标注线标注尺寸和间距，手动切图、压缩，然后打包 ZIP 发给开发。这个流程耗时且极易出错——改一个像素就要重新出图、重新标注、重新打包。如今，Figma Dev Mode、Zeplin、Sketch Inspect 等工具让开发可以直接查看属性，无需设计师手动标注。

Figma Dev Mode（2023 年推出）提供开发者视角，自动显示 CSS、SwiftUI、Compose 代码片段，支持间距测量和资产导出，大幅减少"切图在哪儿？这个颜色值是什么？"这类来回沟通。开发者在 Dev Mode 中点击任何元素都能看到它的尺寸、间距、颜色值和字体属性，甚至可以直接复制代码片段。但工具不能替代一切：工具的代码输出是参考性质的——它不知道你的项目用的是 rem 还是 px，不知道哪些值是全局变量而非硬编码。标注信息仍需要设计师的审核和补充。工具给你信息，但不能替你思考。

## 开发者需要的标注信息

开发者接收设计稿时需要看到以下信息：

1. **尺寸 (Dimensions)**：宽度、高度、最大/最小尺寸、比例约束。每个容器在不同断点下的尺寸变化要明确——移动端是全宽，桌面端是 1200px 居中，这种信息如果不在设计稿中体现，开发就会猜测。

2. **间距 (Spacing)**：内边距 (padding)、外边距 (margin)、元素间距 (gap)。在 Figma 中按住 Alt 悬停即可看到距离，Dev Mode 中可直接测量。间距的一致性比绝对值更重要——如果一个页面用 16px 间距，另一个用 15px，开发者不知道哪个是对的，大概率两个都不会还原对。

3. **颜色 (Colors)**：hex 值（如 `#3B82F6`）、rgba（带透明度场景）、语义变量名（`--color-primary` 而非 `#3B82F6`）。使用 Figma Variables 和 Styles 可以实现一键导出 Design Tokens。关键：开发需要的是变量名而非裸色值——如果你的设计系统中有 `primary-500`，就不要在标注中写 `#3B82F6`。

4. **排版 (Typography)**：字体名称、字号（px/rem）、字重、行高 (line-height)、字间距 (letter-spacing)、段落间距。建议使用 Figma Text Styles 统一管理并导出，避免某个页面的正文是 14px/1.5，另一个页面是 15px/1.6 的不一致。

5. **交互状态**：必须在设计稿中展示 hover、active/pressed、disabled、loading、error、success、empty 等全部状态。只交付一个 default 状态是设计交付中最常见的失误——这等于把交互状态的设计决策推给了开发者，而"disabled 状态用什么灰色"、"hover 时颜色变多深"这些决策应当由设计师来做，不是开发者。

6. **边界情况**：超长文本处理方式（用户名叫"亚历山大·弗拉基米罗维奇·某某夫"时显示不全怎么办——截断还是换行？）、空状态（无数据时显示什么——空白页还是一句引导文案加插画？）、极端图片比例（1:1 的头像和 16:9 的封面图同时出现在列表中的处理规则）。

## 切图与资源导出规范

**格式选择**：SVG 用于图标、logo、简单插画——矢量格式，任意缩放不失真，体积小，是首选格式。PNG 用于需要复杂阴影、渐变、透明度且 SVG 无法完美呈现的场景，导出 @2x 和 @3x 两套以支持不同像素密度。WebP 用于照片和复杂图片——比 PNG 小约 26%，比 JPEG 小约 25-35%，但需要提供 JPEG/PNG 降级方案以兼容旧浏览器。PDF 用于打印场景或需要无损矢量存档时。注意事项：不要用 JPEG 导出图标或带透明度的图形——JPEG 不支持透明，会在透明区域填充白色。也不要在 SVG 中嵌入位图，这违背了使用 SVG 的初衷。

**命名规范**：格式为 `[类型]-[名称]-[尺寸].[扩展名]`，示例包括 `icon-home-24.svg`、`logo-company.svg`、`bg-hero@2x.webp`、`illus-empty-state.svg`。原则：全小写、用连字符不用下划线、描述性而非数字编号、保持一致性。全员使用同一套命名规范，避免有人用 `ico_home.png`、有人用 `home-icon.svg`、有人用 `icon-home.svg` 的混乱局面——混乱的命名会导致开发引用错资源，排查起来极其耗时。

**导出方式**：Figma 中选中图层，在 Export 面板设置格式和倍率后导出，支持批量导出。使用 Slice 工具划定导出区域——当需要导出多个图层合并后的结果时，Slice 比逐个导出再请开发拼接更准确。对于需要频繁更新的资源（如电商类产品每周更换的运营 banner），可配置 CI/CD 自动导出流程（Figma API + GitHub Action），确保切图资源始终与设计稿同步。

## 图层组织规范

好的图层组织让交付更顺畅。**有意义的命名**：把"Rectangle 47"改为"bg-hero-section"，把"Group 12"改为"card-product"。每个图层名都应描述其功能和内容，而不是 Figma 自动生成的编号——当你把文件交给开发者时，"Rectangle 47"传达的信息为零。**逻辑分组**：按功能模块分组（Hero → Features → Testimonials → Footer），不要按图层类型分组——把所有 Rectangle 放一起对理解页面结构毫无帮助，开发者打开文件后首先要花大量时间重新理解结构。

**清理未使用的图层**：隐藏的备用图层、实验性的多个版本、废弃的组件实例——这些都会混淆开发者，让他们不确定哪些图层需要实现，哪些可以忽略。交付前做一次图层清理是必须的。**Frame 标识**：使用有辨识度的 Frame 名称和封面缩略图，让开发者一眼找到需要的页面——把 Frame 名称命名为"Homepage"、"Settings"、"Checkout - Step 2"，而不是"Frame 1"、"Frame 2"。

## Design Tokens：设计的可编程接口

Design Tokens 连接设计决策与代码实现。定义是将设计决策（颜色、字体、间距、阴影等）抽象为平台无关的命名值。例如：`color-primary-500: #3B82F6` 可转换为 CSS 的 `--color-primary-500`、Swift 的 `Color.primary500`、Android 的 `<color name="primary_500">`。工具方面，Style Dictionary（亚马逊开源）负责跨平台转换，Tokens Studio for Figma 作为 Figma 插件管理和导出 tokens，Figma Variables 在 Figma 内定义和可视化编辑。

流程：在 Figma Variables 中定义 → 导出 JSON → 通过 Style Dictionary 转换 → 输出为 iOS、Android、Web 各平台的原生格式。完整的 token 体系应该包括颜色 (color)、间距 (spacing)、字体 (typography)、阴影 (shadow)、圆角 (border-radius)、动效参数 (duration/easing)。价值在于一次定义、多平台同步。当主色从蓝变绿时，只需在 Figma Variables 中修改一次，导出的 tokens 就能自动更新所有平台，彻底消除手动同步带来的版本不一致问题。

## 协作沟通策略

交付是对话，不是甩包。**设计标注文档**：除了 Figma 链接，还需要一份简洁的文档说明页面结构、交互逻辑和边界条件。不需要长篇大论，但要覆盖关键决策和容易遗漏的细节——比如"这个页面使用的是键盘的模式 A 而非模式 B"、"这个过渡动画使用了 Emphasized 缓动 300ms"。**Figma 内注释**：对复杂交互、动画曲线、特殊状态，在 Figma 内添加 Comment 或独立的标注图层。文字说明比指望开发者"看图悟道"可靠得多——一张图可能表达一千个字，但你永远不知道开发者能读到的是哪一百个。

**交付会议**：安排 10-20 分钟的同步会议，设计师 walk through 关键交互和容易误解的地方，开发者现场提问。这比几十条图上的来回评论有效得多——一次 15 分钟的对话能消除 15 天的异步澄清。交付会议不是设计宣讲，而是双向沟通时间。

**交付检查清单**：
1. 是否覆盖所有交互状态（hover / active / disabled / loading / error / empty）？
2. 是否标注了所有间距和尺寸（包括最小/最大约束）？
3. 是否导出了所有切图资源（多倍率、多格式、命名是否符合规范）？
4. 是否定义了动画参数（时长、缓动曲线、延迟 stagger）？
5. 是否标注了键盘导航和屏幕阅读器行为（Tab 顺序、aria-label 内容）？
6. 是否处理了边界情况（超长文本、空数据、错误、网络中断）？

**交付后的跟进**：在开发过程中保持开放沟通，快速响应开发者的疑问。时间越久，偏离设计的实现就越难修正。设计评审（Design QA）——对比开发实现和设计稿，逐屏逐组件找出差异并及时修正——应该在每个迭代内尽快完成，而不是等到发版前才做。良好的交付不是一次性行为，而是设计和开发之间持续的协作关系。
