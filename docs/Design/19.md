# 暗黑模式 (Dark Mode) 设计指南

暗黑模式已从"极客专属"的偏好演变为主流产品的标配功能。从 iOS 的 Dark Appearance 到 macOS 的 Dark Mode，从 VS Code 到微信——用户期望每个产品都提供暗色主题。但暗黑模式绝不是"把白色背景换成黑色"那么简单。

## 为什么需要暗黑模式

三个核心驱动力。**减轻眼部疲劳**：在低光环境中，白色背景的眩光会加速眼睛疲劳。暗色背景减少屏幕发出的光量，让夜间使用更舒适。这不是主观偏好——研究表明在暗环境中，亮色界面确实会引发更多的不适感和眼部干涩，尤其对有干眼症或光敏感的用户。**OLED 屏幕省电**：在 OLED 和 AMOLED 屏幕上，黑色像素完全不发光（像素关闭 = 不耗电）。Google 的测试显示，YouTube 使用暗色主题在 OLED 屏幕上可节省 15-60% 的电量，具体取决于亮度设置——亮度越高，省电效果越显著。对于移动设备用户，这意味着有意义的续航延长。**审美偏好**：暗色主题给人"专业感"和"酷感"。对开发工具（IDE）、设计工具、影音娱乐类产品，暗色主题与内容本身形成更好的对比，让彩色内容成为焦点而灰色界面框架退入背景。这也是为什么照片编辑、视频剪辑和代码编辑软件的默认主题通常是暗色的。

## 暗黑模式不等于纯黑色

这是暗黑模式设计中最常见的误区。不要使用 `#000000` 作为背景色。纯黑在 OLED 上是真正的零亮度，会引发两个问题：一是涂抹效应 (Smearing)，OLED 像素从完全关闭切换到发光状态有微量延迟，滚动时亮色文字在纯黑背景上产生"拖影"——就像用橡皮擦在纸上拖出一道痕迹；二是对比度过高，纯黑上的白色文字非常刺眼，长时间阅读后反而比亮色模式更疲劳，因为瞳孔在巨大的亮度差之间来回调节。

Material Design 推荐的暗色表面：最低层 `#121212`，各层级递进使用 `#1E1E1E`、`#242424`、`#2C2C2C`。Apple 的 Human Interface Guidelines 同样使用深灰黑而非纯黑作为系统底色——iOS 的暗色模式背景色并非 `#000000`。经验值：背景色在 `#121212` 到 `#1F1F1F` 之间，这个范围内的颜色看起来是"黑"的但保留了微弱的发光，既提供暗色体验又保持视觉舒适度。

## 暗黑模式中的层级表达

在亮色模式中，投影 (box-shadow) 用来表达层级——阴影越深越大，说明元素离背景越远、层叠越高。在暗色模式中，黑色阴影融入黑色背景完全看不见，需要用亮度来替代投影的语言。**底层（背景）**：最暗——`#121212`。**第一层（卡片、列表项）**：略亮——`#1E1E1E`。**第二层（弹出菜单、下拉选项）**：更亮——`#242424`。**第三层（对话框、模态窗口）**：继续提亮——`#2C2C2C`。**最高层（通知气泡、工具提示）**：最亮——`#373737` 或更亮。

背后的原理是物理直觉：离光源更近的表面更亮。暗黑模式利用这个直觉——"更亮的表面看起来更近"——来构建空间层次。这就是为什么 Material Design 的 dark theme elevation overlay 使用半透明白色叠加：高层级组件 = 更多半透明白光叠加 = 更亮的表面。每一层约增加 8-12% 的白光不透明度。这个机制让用户在暗黑模式中没有投影的情况下依然能区分组件的层级关系。

## 颜色的适应性调整

颜色在暗色背景上的感知与亮色背景完全不同——同一个色值在两种背景下看起来像两个不同的颜色。**降低饱和度**：同一个颜色在暗色背景上会显得更鲜艳、更刺眼。`#3B82F6`（亮蓝）在白色背景上舒适，但当背景变为 `#121212` 时，它看起来像在"发光"——暗背景吸收了周围的光，让亮色显得更加突出。解决方案是在暗色模式中将饱和度降低 20-40%。Material Design 建议对 primary color 使用 200 号色调（更柔和的版本）而非 500 号——这是经过大量实验得出的经验值。

**调整亮度**：浅色在暗色背景上更明显。原来在亮色模式中用作边框或分隔线的中灰色(`#E0E0E0`)，在暗色模式中可能需要提亮到 `#383838` 才能达到相同的"视觉权重"。**避免高饱和纯色**：红色 (`#FF0000`)、纯绿 (`#00FF00`)、纯蓝 (`#0000FF`) 在暗黑模式中像霓虹灯一样刺眼——它们与暗背景的对比度极高，视觉上会呈现"光晕"效果。使用柔和色调替代——错误红用 `#CF6679`（Material 推荐），成功绿用 `#4CAF50` 而非 `#00FF00`。

**品牌色处理**：品牌色需要在暗色模式中保持辨识度但不能刺眼。通常做法是提亮品牌色的 L 值（HSL 中的 Lightness）或降低 S 值（Saturation），在暗背景上寻找一个可辨识但舒适的变体。这个变体不需要和亮色模式中的值有任何数学关系——它只需要在暗背景上"看起来像"品牌色。

## 文字在暗色背景上的处理

文字是暗黑模式中最难处理好的部分，也是区分"随便做了一个暗色模式"和"认真设计了暗色模式"的关键。不要使用 `#FFFFFF` 纯白文字——在暗色背景上纯白文字过于刺眼，因为人眼在暗环境中瞳孔放大，接收到的光量增加，纯白会"灼伤"视网膜。推荐正文颜色：`rgba(255, 255, 255, 0.87)` 或 `#E0E0E0`——略微偏灰的白，既保留足够的对比度又不过度刺眼。次要文字：`rgba(255, 255, 255, 0.60)` 或 `#9E9E9E`——提供信息层级但不抢夺注意力。禁用和占位文字：`rgba(255, 255, 255, 0.38)` 或 `#757575`——明确传达"不可用"但仍在可辨识范围内。高亮和强调文字：`rgba(255, 255, 255, 0.95)`——比正文略亮以示强调，但避免纯白。

需要检查对比度：这些透明度值对应的对比度在 `#121212` 背景上需不低于 4.5:1（AA 级）。0.87 不透明度的白色在 `#121212` 上约为 15:1，远超要求但视觉上仍舒适，因为灰色背景吸收和散射了部分白光，降低了刺眼感。这是暗黑模式文字设计的微妙之处——对比度数值足够高，但视觉感受不刺眼。这和亮色模式正好相反——亮色模式中 15:1 的对比度（纯黑文字在白色背景上）就是刺眼的。

## 系统适配与用户控制

**自动跟随系统**：使用 CSS 媒体查询 `prefers-color-scheme: dark` 自动检测系统主题。这是最基础也是最优选的实现方式——大多数用户期望 App 自动匹配系统设置，日落时系统自动切换到暗色模式，App 应同步响应。

**手动切换**：同时提供手动 toggle，三个选项——浅色、暗色、跟随系统。允许用户覆盖系统设置以满足个性化需求。有些用户在白天也偏好暗色模式（如长期在暗环境中工作的开发者），有些用户在夜间反而偏好亮色（如年长用户在暗色下阅读困难）。"跟随系统"应为默认选项，"浅色"和"暗色"是用户主动选择后的覆盖。

**持久化选择**：将用户的选择存储在 localStorage（Web）或 UserDefaults（iOS）和 SharedPreferences（Android），下次访问时记住偏好，避免每次都要重新切换。注意处理初始化竞态——在页面加载的第一帧就应用正确的主题，而不是先闪一下默认主题再切到用户偏好的主题。Flash of Wrong Theme (FOWT) 是暗黑模式实现中最常见的问题。

**切换过渡**：在亮暗切换时添加 200-400ms 的过渡动画，避免突兀的跳变。不过需要注意，浏览器的 `prefers-color-scheme` 切换无法直接使用 CSS transition 平滑过渡——系统切换主题时浏览器会重新计算所有样式而非执行 transition。需要使用 JavaScript 做颜色插值或使用 CSS 自定义属性配合 View Transitions API (`document.startViewTransition`) 来实现平滑切换。

## 暗黑模式特有的设计挑战

**投影变成发光**：在亮色模式中 `box-shadow` 清晰可见，在暗色模式中黑色阴影完全融入黑色背景。解决方案是将投影替换为微弱的辉光（使用亮色而非暗色的 box-shadow 模糊扩散，模拟发光边缘）或完全依赖亮度层级来表达深度（前述 elevation overlay 方法）。

**透明背景的图片**：为亮色模式设计的 logo 或插画在白色底上显示完好，但放在暗色背景上时，logo 中的深色部分就"消失"了。解决方案：准备两套资源（浅色版和深色版），或使用 SVG 配合 `currentColor` 和 CSS 变量实现自适应——SVG 内部的填充色通过 CSS 变量指定，亮暗切换时自动换色。

**表单和输入框**：白色输入框在暗色背景上形成强烈反差，破坏视觉一致性。应使用深色背景（如 `#1E1E1E`）加灰色边框（`#555`）来区分输入框与背景，聚焦时边框变亮为品牌色。**选中文本**：`::selection` 的背景色和文字色在两种模式下都应有足够的对比度——不要用系统默认的蓝色选择背景配黑色文字，在暗色下完全看不清。**滚动条**：暗色背景加暗色滚动条等于看不见滚动条，用户无法判断页面长度。需要自定义滚动条样式或至少确保浏览器默认滚动条在暗色模式下有足够的对比度（Webkit 提供了 `scrollbar-color` 属性）。

## Figma 中的暗黑模式设计

不要为暗黑模式单独维护一套设计稿——那意味着每次改动都要同步两个文件，出错只是时间问题。使用 Figma Variables 的 Mode 功能：同一个 Variable（如 `color/bg/surface`）定义 Light 和 Dark 两个 Mode 的值，组件引用变量后在两种模式下自动切换颜色。使用 Swap Library 或 Theme Switcher 插件一键切换整个文件的亮暗预览。导出 Design Tokens 时，Light 和 Dark 两套值一起导出为同一个 JSON 文件，方便开发实现主题切换逻辑。关键是：变量命名应该是语义化的（`color/bg/surface`）而非外观描述的（`color/gray-900`），因为同一个语义变量在亮暗模式下可能是完全不同的颜色值。

## 测试清单

正式发布暗黑模式前，逐项验证：

- [ ] 所有文字在暗色背景上可读，对比度不低于 4.5:1（AA 级）
- [ ] 重点色已根据暗色背景调整饱和度和亮度，看起来是"同一个品牌色"而非"亮瞎眼的霓虹色"
- [ ] 投影和层级关系在暗色模式中依然清晰可辨（通过亮度而非阴影区分）
- [ ] Logo 和图标在暗色背景下每个像素都可见，深色元素未"消失"
- [ ] 表单组件、按钮、输入框在两种模式下都可用且视觉一致
- [ ] 亮暗切换过渡流畅，无闪烁或布局跳动，无 FOWT (Flash of Wrong Theme)
- [ ] 用户的手动主题选择被正确保存，下次访问时恢复上一次偏好
- [ ] 第三方嵌入内容（支付组件、地图、社交媒体嵌入）的暗色适配已确认或已提供替代方案
- [ ] 邮件模板和其他脱离 App 的内容在暗色背景下不会因暗色文本而无法阅读
- [ ] 所有页面和状态（包括错误页、404 页、加载态）都已在暗色模式下验证
