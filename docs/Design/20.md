# 表单设计最佳实践

表单是用户与产品之间最重要的对话界面。注册、登录、结账、搜索、设置——每一条用户输入都是信任的交换。一个差的表单设计会让用户放弃注册、放弃购物车、甚至放弃整个产品。表单设计的目标不是"收集信息"，而是"帮助用户完成填写"。

## 单列布局：垂直扫描法则

表单布局的第一原则就是单列:

- 用户在填写表单时从上到下垂直扫描，多列布局会打断这个流
- 多列的问题: 用户可能跳过一个字段（以为已经填过了），理解顺序混乱（先填左边还是上边？），在移动端完全不可用
- 仅有的例外: 语义上天然成对的字段可以放在一行——城市/省份、信用卡号/有效期/CVV
- 但即使是"成对"字段，在移动端也应该转为单列。320px 宽的屏幕上放两列输入框，每个输入框不足 140px，无法正常显示 placeholder
- 从视觉流的角度，单列布局建立了一个清晰的"输入路径"，用户可以专注于填写而非理解布局

## 标签的位置之争

Label placement 直接影响扫描速度和完成时间:

- **顶部标签 (Top-aligned)**: 标签在输入框上方。扫描最快，完成速度最快。对于任何形式的表单（包括复杂表单）都是推荐的做法。用户从上到下扫描——读标签——填写——继续向下，一气呵成。
- **左侧标签 (Left-aligned)**: 标签在输入框左侧。阅读标签和输入框之间的眼跳距离长，填写速度最慢（比顶部标签慢约 50%）。唯一优势: 垂直空间紧凑。适用于字段较少的工具栏或筛选面板。
- **右侧标签 (Right-aligned)**: 标签在输入框右侧（右对齐）。标签长度不一导致起始位置不整齐。极少使用，仅在特殊场景。
- **浮动标签 (Floating Labels)**: Material Design 推广的模式——placeholder 作为标签，聚焦时缩小并上移。优点: 节省垂直空间，标签永久可见（不像 placeholder 标签），在移动端表现良好。争议点: 可用性研究结果不一，小字号（10-12px）的浮动标签可能难以阅读，长标签会溢出。Material Design 3 中已弱化了浮动标签的强制性。

## 输入框大小匹配预期内容

输入框宽度传递了"答案应该多长"的信号——这是一种无声的微交互设计:

- 邮编输入框: 短（容纳 6-10 个字符即可）
- 地址输入框: 长（容纳完整地址）
- 信用卡号: 刚好容纳 19 个字符（16 位数字 + 3 个空格）
- 不要在邮编输入框上分配全宽度，这会给用户错误的预期
- 常见错误: Web 表单上所有输入框使用统一的 100% 宽度。这是一种开发者思维（方便布局），而非设计思维（以用户为中心）

## Placeholder 不是 Label

这是表单设计中最根深蒂固的反模式:

- Placeholder 用作文本格式示例（如 "example@email.com"、"4000 1234 5678 9010"），这是正确的用法
- Placeholder 决不应该替代标签。原因有三:
  1. 用户一开始输入，placeholder 就消失了——用户忘记这个字段要填什么
  2. Placeholder 的默认灰色通常对比度不达标（尤其是亮色主题下）
  3. 屏幕阅读器可能忽略 placeholder 或将其与 label 混淆
- 如果标签和输入框之间的空间让你觉得"浪费"，解决问题的方法是重新审视表单整体设计（合并字段、使用条件显示），而不是用 placeholder 省去标签

## 校验策略：时机决定体验

三种校验时机，各有用武之地:

- **即时校验 (Inline Real-time)**: 用户在输入过程中就得到反馈。适用于: 用户名可用性检查、密码强度指示、格式明确的字段（如邮箱格式）。绿色对勾给人满足感，红色警告即时阻止错误。
- **失焦校验 (On Blur)**: 用户离开字段时校验。适用于: 需要完整输入后再判断的校验（如两次密码是否一致），或输入过程中频繁显示错误会影响体验的场景。
- **提交校验 (On Submit)**: 点击提交按钮时统一校验。适用于: 相对复杂的校验规则、后端校验。这是最后的防线，但不能是唯一的校验方式。
- 最佳实践: 三种方式结合。即时校验用于前置引导（帮用户正确填写），失焦校验用于不复杂的规则，提交校验作为最终把关。
- 关键规则: 提交校验失败后，焦点必须移动到第一个出错的字段，且所有错误以汇总形式在表单顶部列出（同时在各字段旁显示）。

## 错误信息与帮助文本

错误信息决定表单体验的成败:

- **具体而非笼统**: "密码至少需要 8 个字符，包含 1 个大写字母和 1 个数字" vs "密码格式错误"。前者告诉用户怎么做，后者让用户猜测。
- **友好的语气**: "请输入有效的邮箱地址" vs "邮箱格式错误"。前者是帮助，后者是指责。
- **帮助文本的位置**: 字段下方，紧邻字段。不要让用户去找错误信息。
- **颜色 + 图标 + 文字**: 不要仅靠红色边框传递错误信息。色盲用户可能看不出颜色变化。错误必须同时有: 红色边框 + 错误图标 + 文字说明。
- **成功也有反馈**: 字段填写正确后给予轻量级的正面反馈（绿色对勾），尤其是在密码强度指示等场景中。

## 必填 vs 可选：二选一的困惑

设计惯例有两种，选一种并保持一致:

- **方案一**: 必填字段用红色星号 (*) + "必填"文字标记。可选字段不加标记。这是最广泛使用的方案，因为"什么是必填"通常比"什么是可选"更重要的信息。
- **方案二**: 可选字段标记"(可选)"。必填字段不加任何特殊标记。这个方案适用于大多数字段都是必填的场景——用户只需要被提醒什么是可以跳过的。
- 常见错误: 把所有字段都标为必填（那就不需要标记了），或者同时为必填和可选都加标记（冗余，增加视觉噪音）。
- 如果表单中必填字段少于可选字段，使用方案二。如果必填字段占大多数，使用方案一。

## 渐进式展示

不要让用户一开始就面对所有字段——逐步揭示复杂度:

- 第一步只展示最核心的必填字段
- 高级选项（如"公司名称"、"发票信息"）默认折叠，点击"更多选项"展开
- 条件字段仅在相关选项被选中时出现。例如：选择了"其他"国籍后，才出现"请描述"文本框；选择了"公司"账户类型后，才出现公司名称输入框
- 这种设计减少了视觉压力——用户面对一个只有 3 个字段的表单比面对 15 个字段的表单更愿意开始填写

## 操作按钮的布局

提交按钮的放置关系到表单的最终转化:

- 主操作按钮（如"提交"、"注册"、"登录"）左对齐放置在表单下方
- 为什么左对齐: 用户的视线在扫描完表单后自然停留在左下角。而且对齐到输入框的左边缘，形成视觉上的连贯
- 为什么不是右对齐: 西方阅读习惯是从左到右，但表单填写不是阅读——是在一列中不断向下。将按钮放在右侧打破了这个流
- 辅助操作（如"取消"、"返回"）作为文字链接或次要按钮放在主按钮右侧
- 提交中状态禁用按钮（防止重复提交），同时显示加载动画（spinner 或文字变为"提交中..."）

## 总结

表单设计的核心原则可以归结为一句话: **让填写变得毫不费力**。单列布局减少认知负担，顶部标签加速扫描，渐进式展示降低开始门槛，精准的校验和友好的错误信息引导用户走向成功。每当你设计一个表单，问自己: 如果一个完全不了解这个产品的人，能在 30 秒内理解这个表单要收集什么、如何填写、以及填完后会发生什么吗？如果答案是"不确定"，那就还有改进的空间。
