# 设计系统 (Design System) 构建方法论

设计系统是产品在规模化发展过程中自然产生的一种必然需求。当你只有一两个页面和一名设计师时，一致性靠的是记忆和沟通。当团队扩展到 10 位设计师、3 个产品和 50 个页面时，靠人脑维持视觉一致性是不可能的任务。设计系统正是这个问题的系统性解决方案：它是一套由设计原则、可复用组件和配套文档组成的工程化体系。

## 设计系统为什么重要

设计系统的价值可以从三个维度来理解。**一致性**是表面价值——用户在不同的功能模块中看到相同的按钮、相同的色彩语言、相同的交互模式，降低了认知负担。**效率**是底层价值——设计师不需要从零画按钮，开发者不需要反复写购物车卡片样式，每个人都站在已构建好的组件基石上去解决更高层的问题。**单一事实来源**是工程价值——当设计规范和代码组件同步维护时，设计与开发之间的沟通成本和交付偏差大幅降低。

## 设计系统的层级结构

一个成熟的设计系统通常从底层到顶层分为四个层级，这套方法论由 Brad Frost 的 Atomic Design 理论发展而来：

### Design Tokens（设计令牌）

最底层的原子单位，是设计系统中所有视觉属性的唯一编码表示。常见的 token 包括：

- **颜色 token**：`color-primary-500` = #3B82F6, `color-neutral-100` = #F5F5F5
- **字体 token**：`font-size-heading-xl` = 32px, `font-family-body` = Inter
- **间距 token**：`spacing-4` = 16px, `spacing-6` = 24px
- **阴影 token**：`shadow-card` = 0px 2px 8px rgba(0,0,0,0.08)
- **圆角 token**：`radius-md` = 8px, `radius-full` = 9999px

Token 化的核心优势在于：当你需要全局调整主色调时，只需修改一处 token 定义，所有引用该 token 的组件都会自动更新。在 Figma 中，你可以使用 **Styles**（颜色样式、文字样式、效果样式）和 **Variables**（Figma 的新一代变量系统，支持模式和主题）来实现 token 管理。

### Atoms（原子组件）

由 Token 组合而成的最小可识别 UI 单位，无法再拆分。典型原子组件包括：Button、Input、Icon、Label、Badge、Avatar、Checkbox。

### Molecules（分子组件）

由多个原子组件组合而成的功能性单元，能够独立承担一个具体的交互任务。例如：SearchBar = Input + Icon + Button；FormField = Label + Input + Error Message；Card = Image + Title + Description + Button。

### Organisms（有机体组件）

由分子和原子共同构成的完整区块，通常对应页面上一个独立的功能区域。例如：Header = Logo + NavLinks + SearchBar + UserMenu；ProductList = FilterBar + ProductCards[ ] + Pagination。

## 从零构建设计系统的路径

### 第一步：审计现有 UI

不要试图从零搭建理想化的完美系统——那是一种陷阱。拿出你的产品现有的所有页面截图，将相似的 UI 元素归类。你会发现 12 种不同的灰色背景、8 种尺寸的按钮、5 种不同的输入框样式。审计的目的不是批评，而是**发现重复模式**——这正是你需要标准化的地方。

### 第二步：定义 Design Tokens

从审计结果出发，提取出一份 token 清单。优先处理使用频率最高、变体最多的属性，通常顺序是：颜色 → 间距 → 字体 → 圆角 → 阴影。在 Figma 中将每种 token 创建为 Style 或 Variable。

### 第三步：构建原子组件

按使用频率从高到低依次构建。Button 和 Input 通常是第一个。每完成一个组件，立即在 Figma 中创建组件变体集，添加各种状态（default/hover/active/disabled/focus）和尺寸（S/M/L）。

### 第四步：编写文档

未经文档化的组件等于不存在。每个组件的文档至少包含：组件名称和用途描述、视觉预览（各变体截图）、使用场景和禁忌（Do/Don't 示例）、对应的代码组件名称（方便开发者对应）、设计文件中的实际路径（方便设计师查找）。

常用文档平台包括：Figma 文件内的组件说明页面（最轻量）、Storybook（代码组件为主，设计师也能访问）、Zeroheight（专门的设计系统文档平台，支持与 Figma 实时同步）。选择哪个平台取决于团队规模和预算。

### 第五步：建立维护机制

设计系统不是一次性交付物，而是需要持续养护的活系统。核心维护实践包括：使用语义化版本命名（v1.0、v1.1、v2.0），每次更新记录 Changelog，建立组件添加和修改的贡献流程（提案 → 评审 → 实现 → 文档 → 发布），明确设计系统的负责人（Design System Lead 或 Design Ops）。

## 业界参考案例

- **Material Design（Google）**：最完整、文档最丰富，适合作为学习范本。MD3 引入了 Dynamic Color 和个性化主题概念。
- **Ant Design（蚂蚁集团）**：中国影响力最大的企业级设计系统，提供了从设计语言到 React 组件库的完整生态。
- **Shopify Polaris**：电商场景的专业设计系统，在文案指引和可访问性方面尤为出色。
- **Carbon（IBM）**：面向企业级复杂系统，在数据可视化组件和数据表格方面有深入实践。

## 一份务实的起步建议

对于小团队（1-5 名设计师），完整的设计系统建设可能过于奢侈。推荐的务实路径是：先从 Figma Styles 统一颜色和文字 → 提取 5-10 个最常用组件做标准化 → 在一个功能模块中试点使用 → 验证效果后再扩展到全产品。关键是先让一致性收益可见，再逐步追加投入。
