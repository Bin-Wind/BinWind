# 定时任务 Crontab 实用指南

Crontab 是 Linux 系统中用于定时执行任务的服务，从日志清理到数据备份，从证书续期到健康检查，几乎所有周期性的自动化操作都依赖它。本文从语法基础讲到生产实践，帮你掌握 crontab 的方方面面。

## Crontab 语法

一条 cron 任务由两部分组成：时间表达式和要执行的命令。

```text
# ┌─────────── 分钟 (0 - 59)
# │ ┌─────────── 小时 (0 - 23)
# │ │ ┌─────────── 日 (1 - 31)
# │ │ │ ┌─────────── 月 (1 - 12)
# │ │ │ │ ┌─────────── 星期 (0 - 7, 0 和 7 都代表周日)
# │ │ │ │ │
# * * * * * command_to_execute
```

### 时间字段详解

| 字段 | 范围 | 特殊字符 |
|------|------|----------|
| 分钟 | 0-59 | `*` `,` `-` `/` |
| 小时 | 0-23 | `*` `,` `-` `/` |
| 日 | 1-31 | `*` `,` `-` `/` |
| 月 | 1-12 | `*` `,` `-` `/` |
| 星期 | 0-7 (0=周日) | `*` `,` `-` `/` |

**特殊字符**：
- `*`：每个时间单位都匹配
- `,`：列举多个值，如 `1,3,5`
- `-`：范围，如 `1-5`
- `/`：步进，如 `*/5` 表示每 5 个单位

## 常用时间表达式

```bash
# 每分钟执行
* * * * * command

# 每 5 分钟执行
*/5 * * * * command

# 每小时执行（整点）
0 * * * * command

# 每天凌晨 2 点执行
0 2 * * * command

# 每天早上 8:30 执行
30 8 * * * command

# 每周日凌晨 3 点执行
0 3 * * 0 command

# 每月 1 号凌晨 4 点执行
0 4 1 * * command

# 每 15 分钟执行（9:00-17:00，周一到周五）
*/15 9-17 * * 1-5 command

# 每 4 小时执行（0:00, 4:00, 8:00, ...）
0 */4 * * * command

# 每年 1 月 1 日执行
0 0 1 1 * command

# 系统启动时执行一次
@reboot /path/to/startup.sh

# 预定义宏
@yearly    # 每年 1 月 1 日 0:00（等同于 0 0 1 1 *）
@monthly   # 每月 1 号 0:00（等同于 0 0 1 * *）
@weekly    # 每周日 0:00（等同于 0 0 * * 0）
@daily     # 每天 0:00（等同于 0 0 * * *）
@hourly    # 每小时整点（等同于 0 * * * *）
```

## 管理 Crontab

### 编辑和查看

```bash
# 编辑当前用户的 crontab（首次会提示选择编辑器）
crontab -e

# 查看当前用户的 crontab
crontab -l

# 删除当前用户的 crontab（全部清除）
crontab -r

# 以其他用户身份管理 crontab（root 权限）
sudo crontab -u alice -e
sudo crontab -u alice -l

# 从文件导入 crontab
crontab cronfile.txt
```

### 编辑器选择

```bash
# 临时指定
EDITOR=vim crontab -e

# 永久设置（添加到 ~/.bashrc 或 ~/.zshrc）
export EDITOR=vim
```

## 输出重定向

Cron 任务的默认行为是将 stdout 和 stderr 通过邮件发送给用户（如果系统安装了 MTA）。在生产环境中，通常需要手动处理输出：

```bash
# 禁止所有输出（丢弃 stdout 和 stderr）
0 2 * * * /backup/script.sh > /dev/null 2>&1

# stdout 追加到日志文件，stderr 也追加到同一文件
0 2 * * * /backup/script.sh >> /var/log/backup.log 2>&1

# stdout 和 stderr 分开记录
0 2 * * * /backup/script.sh >> /var/log/backup.log 2>> /var/log/backup.err

# 仅记录错误
0 2 * * * /backup/script.sh > /dev/null 2>> /var/log/backup.err
```

**命令解读** `> /dev/null 2>&1`：
- `> /dev/null`：将 stdout 重定向到黑洞
- `2>&1`：将 stderr (2) 重定向到 stdout (1) 当前指向的位置（即 `/dev/null`）

## Crontab 中的环境问题

Cron 执行环境与交互式 Shell 不同，这是最常见的踩坑点：

### PATH 变量有限

Cron 的默认 PATH 通常只包含 `/usr/bin:/bin`，很多命令会找不到：

```bash
# 查看 cron 的 PATH
* * * * * echo $PATH > /tmp/cron_path.txt

# 解决方式一：在 crontab 开头设置 PATH
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

# 解决方式二：脚本中使用绝对路径
0 2 * * * /usr/local/bin/python3 /opt/scripts/backup.py
```

### 工作目录不确定

Cron 默认的工作目录通常是用户的主目录（`$HOME`），而不是脚本所在目录：

```bash
# 错误：依赖相对路径可能导致找不到文件
0 2 * * * cd /opt/app && ./backup.sh

# 正确：使用绝对路径，或在脚本中先 cd
0 2 * * * /opt/app/backup.sh
```

### 最佳实践：在脚本中封装

将复杂逻辑写在独立的 Shell 脚本中，crontab 只负责调度：

```bash
# crontab 中只写调度和脚本路径
0 2 * * * /opt/scripts/daily_backup.sh
```

```bash
#!/bin/bash
# /opt/scripts/daily_backup.sh
# 在脚本中处理好所有环境问题

set -e                      # 任何命令失败立即退出
export PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
export LANG=en_US.UTF-8

LOG_FILE="/var/log/daily_backup_$(date +%Y%m%d).log"

# 记录开始时间
echo "[$(date)] 备份开始" >> $LOG_FILE

# 切换到脚本所在目录
cd "$(dirname "$0")" || exit 1

# 备份逻辑
/usr/bin/mysqldump -u backup -p'password' mydb | gzip > "/backup/mydb_$(date +%Y%m%d_%H%M).sql.gz" 2>> $LOG_FILE

# 清理 7 天前的备份
find /backup -name "mydb_*.sql.gz" -mtime +7 -delete

echo "[$(date)] 备份完成" >> $LOG_FILE
```

## 系统级 Crontab

除了用户级 crontab（`crontab -e`），Linux 还提供系统级的定时任务机制：

### /etc/crontab

系统级 crontab 文件，比用户级多了"执行用户"字段：

```text
# /etc/crontab
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/sbin:/bin:/usr/sbin:/usr/bin

# m   h  dom mon dow  user    command
  0   2  *   *   *    root    /opt/scripts/system_backup.sh
  */15 *  *   *   *    www-data /usr/bin/php /var/www/html/cron.php
```

### /etc/cron.d/

目录，每个文件是一条独立的系统级 cron 配置（格式同 `/etc/crontab`）：

```bash
# 文件：/etc/cron.d/nginx-cert-renew
# 每天凌晨 3 点检查并续期 Let's Encrypt 证书
0 3 * * * root certbot renew --quiet --post-hook "systemctl reload nginx"
```

### /etc/cron.hourly/、/etc/cron.daily/、/etc/cron.weekly/、/etc/cron.monthly/

将可执行脚本放入对应目录，系统通过 `/etc/crontab` 中的 run-parts 机制定时执行：

```bash
# 查看 run-parts 的调度配置
grep -E "cron\.(hourly|daily|weekly|monthly)" /etc/crontab
# 示例输出：
# 17 *    * * *   root    cd / && run-parts --report /etc/cron.hourly
# 25 6    * * *   root    test -x /usr/sbin/anacron || { cd / && run-parts --report /etc/cron.daily; }
```

**使用方式**：

```bash
# 将备份脚本放到 /etc/cron.daily/
sudo cp my_backup.sh /etc/cron.daily/
sudo chmod +x /etc/cron.daily/my_backup.sh
```

**重要**：放置到这些目录的脚本**不能有扩展名**（run-parts 默认忽略包含 `.` 的文件名）。Debian/Ubuntu 的 `--lsbsysinit` 选项无此限制，但跨发行版最好还是去掉扩展名。

## 调试技巧

### 1. 先手动执行

在任何 cron 设置之前，先用与 cron 相同的环境手动执行命令：

```bash
# 模拟 cron 的最小环境运行脚本
env -i HOME=$HOME SHELL=/bin/bash PATH=/usr/bin:/bin \
    /opt/scripts/myscript.sh
```

### 2. 添加调试输出

```bash
# 临时添加调试任务
* * * * * /opt/scripts/myscript.sh >> /tmp/cron_debug.log 2>&1
# 观察输出后移除或调整为实际频率
```

### 3. 检查 cron 日志

```bash
# 查看 cron 服务日志
sudo grep CRON /var/log/syslog              # Debian/Ubuntu
sudo journalctl -u cron                     # 使用 systemd 的发行版
sudo journalctl -u cron -f                  # 实时跟踪
sudo journalctl -u cron --since "1 hour ago"

# 查看 cron 守护进程状态
sudo systemctl status cron
```

### 4. 常见错误排查

```bash
# 检查 crontab 语法是否正确
# crontab -e 保存时会自动验证，有语法错误会提示

# 确认脚本有执行权限
ls -la /opt/scripts/myscript.sh
chmod +x /opt/scripts/myscript.sh

# 确认路径存在
test -f /opt/scripts/myscript.sh && echo "OK"

# 确认 cron 守护进程在运行
pgrep cron
sudo systemctl status cron
```

## Anacron — 非 24/7 系统的补充

Cron 假设系统 7x24 小时运行。对于桌面电脑或非持续在线的服务器，如果计划任务的时间点恰好处于关机状态，任务就会被跳过。Anacron 解决了这个问题：

```bash
# 安装（通常已预装）
sudo apt install anacron

# Anacron 配置：/etc/anacrontab
# 格式：period delay job-identifier command
# period: 单位是天; delay: 系统启动后延迟多少分钟执行
```

```text
# /etc/anacrontab 示例
1       5       cron.daily      run-parts --report /etc/cron.daily
7       10      cron.weekly     run-parts --report /etc/cron.weekly
30      15      cron.monthly    run-parts --report /etc/cron.monthly
```

Anacron 通常与 cron 配合工作：cron 负责小时级及更频繁的任务，Anacron 确保每日/每周/每月的任务即使在关机期间错过也能在下次开机后补执行。

## 生产环境建议

1. **所有路径使用绝对路径**：命令、脚本、日志文件路径全部使用绝对路径
2. **复杂逻辑封装为独立脚本**：crontab 中只保留调度信息，业务逻辑在脚本中
3. **添加日志记录**：每个脚本记录开始时间、结束时间和执行结果
4. **使用 `flock` 防止并发执行**：
   ```bash
   # 如果上次执行尚未完成，跳过本次
   0 * * * * flock -n /tmp/myjob.lock /opt/scripts/myjob.sh
   ```
5. **设置错误通知**：关键任务失败时发送告警：
   ```bash
   0 2 * * * /opt/scripts/backup.sh || \
       echo "备份失败 at $(date)" | mail -s "备份告警" admin@example.com
   ```
6. **设置 MAILTO 变量**：有邮件系统时，cron 会将输出自动发送到指定邮箱
   ```bash
   MAILTO=admin@example.com
   0 2 * * * /opt/scripts/backup.sh
   ```
7. **定期审计 crontab**：确保没有过时的或未记录在案的任务在运行：
   ```bash
   # 审计所有用户的 crontab
   for user in $(cut -f1 -d: /etc/passwd); do
       echo "=== $user ==="
       sudo crontab -u $user -l 2>/dev/null
   done
   ```
