# 日志管理与分析

日志是服务器运维的眼睛和耳朵。无论是排查故障、审计操作还是性能分析，都离不开系统日志。本文将围绕 `journalctl` 和 `logrotate` 两个核心工具，介绍日志的收集、查询、轮转与集中管理方案。

## journalctl 基础查询

`journalctl` 是 systemd 自带的日志查看工具，读取二进制的 journal 文件，查询速度快且支持丰富的过滤条件。

```bash
# 查看所有日志（从旧到新）
journalctl

# 从最新开始查看
journalctl -r

# 实时跟踪日志（类似 tail -f）
journalctl -f

# 限制显示行数
journalctl -n 50

# 按优先级过滤（0=emerg, 1=alert, 2=crit, 3=err, 4=warning, 5=notice, 6=info, 7=debug）
journalctl -p err        # 只看 error 及以上
journalctl -p 3          # 等价写法
```

## 按服务和时间过滤

```bash
# 查看指定服务的日志
journalctl -u nginx
journalctl -u sshd -f   # 实时跟踪 SSH 日志

# 时间范围过滤
journalctl --since "2025-06-01"
journalctl --since "2025-06-01" --until "2025-06-02"
journalctl --since "30 min ago"
journalctl --since "1 hour ago"
```

## 常用查询场景

```bash
# 查看本次启动的日志
journalctl -b

# 查看上一次启动的日志（排查重启前的问题）
journalctl -b -1

# 查看内核消息（相当于 dmesg）
journalctl -k

# 查看特定服务的错误日志
journalctl -u nginx -p err --since today

# 查看特定可执行文件的日志
journalctl /usr/bin/sshd
```

## 持久化 Journal 配置

默认情况下，journal 日志存储在内存中（/run/log/journal），重启后丢失。修改配置实现持久化：

```conf
# /etc/systemd/journald.conf
[Journal]
Storage=persistent          # 持久化到磁盘
SystemMaxUse=500M           # 日志最大占用空间
SystemKeepFree=1G           # 保留的空闲磁盘空间
MaxRetentionSec=2week       # 最大保留时间
Compress=yes                # 启用压缩
```

修改后重启服务：

```bash
sudo systemctl restart systemd-journald
```

日志文件将存储在 `/var/log/journal/` 目录下，重启后不会丢失。

## logrotate 日志轮转

对于不使用 journal 的应用（如 Nginx、MySQL），日志轮转需要靠 `logrotate` 来完成。主配置文件为 `/etc/logrotate.conf`，各应用的独立配置放在 `/etc/logrotate.d/` 目录下。

```conf
# /etc/logrotate.d/nginx
/var/log/nginx/*.log {
    daily                   # 每天轮转一次
    missingok               # 日志文件不存在也不报错
    rotate 30               # 保留 30 个归档文件
    compress                # 压缩归档日志
    delaycompress           # 延迟压缩（保留最近一个不压缩）
    notifempty              # 日志为空则不轮转
    maxage 30               # 超过 30 天的归档删除
    dateext                 # 使用日期作为后缀
    dateformat -%Y%m%d      # 日期格式
    postrotate
        # 向 Nginx 发送 USR1 信号，重新打开日志文件
        [ -f /var/run/nginx.pid ] && kill -USR1 $(cat /var/run/nginx.pid)
    endscript
}
```

常用轮转策略：
- **按大小**：`size 100M` — 日志达到 100MB 时轮转
- **按周期**：`daily` / `weekly` / `monthly`
- **maxage**：超过指定天数的归档强制删除
- **compress** + **delaycompress**：压缩但保留最近一份未压缩，方便查看

手动测试 logrotate 配置：

```bash
# 强制执行一次轮转（调试模式，不实际修改）
sudo logrotate -d /etc/logrotate.conf

# 强制执行轮转
sudo logrotate -f /etc/logrotate.conf
```

## 集中日志管理：rsyslog 远程转发

当服务器数量增多时，逐台查看日志效率极低。通过 rsyslog 将日志转发到中央日志服务器统一管理。

在客户端配置转发：

```conf
# /etc/rsyslog.d/50-remote.conf
# 将所有日志通过 TCP 发送到中央日志服务器
*.* @@192.168.1.100:514
```

在日志服务器端开启接收：

```conf
# /etc/rsyslog.d/50-server.conf
module(load="imtcp")
input(type="imtcp" port="514")
```

更高级的方案是使用 ELK（Elasticsearch + Logstash + Kibana）或 Loki + Grafana 搭建日志平台，支持全文搜索和可视化面板，适合生产环境下的大规模日志分析。

## 小结

`journalctl` 是 systemd 服务的首选日志工具，配合持久化配置可以覆盖大部分场景。对于第三方应用的日志，`logrotate` 提供了灵活的轮转策略。当日志规模增长到多台服务器时，集中日志管理是提升运维效率的关键一步。
