# 进程监控与故障排查

服务器运行缓慢、端口被占用、程序异常退出，这些都离不开进程级别的排查工具。本文介绍 `htop`、`lsof`、`strace` 和 `ps` 等常用进程监控与调试工具的使用方法，帮助你快速定位线上问题。

## htop：交互式进程管理器

`htop` 是 `top` 的增强版，提供彩色界面和交互式操作。首先确保已安装：

```bash
sudo apt install htop        # Debian/Ubuntu
sudo yum install htop        # CentOS/RHEL
```

常用快捷键：

| 快捷键 | 功能 |
|--------|------|
| F3 / `/` | 搜索进程 |
| F4 | 按进程名过滤 |
| F5 | 进程树视图 |
| F6 | 选择排序字段 |
| F9 | 发送终止信号 |
| F10 / `q` | 退出 |
| `t` | 切换进程树模式 |
| `H` | 切换线程显示 |
| `u` | 按用户过滤 |

运行 `htop` 后，顶部进度条直观展示 CPU 和内存使用率。按 `F6` 可以选择按 CPU 占用（PERCENT_CPU）或内存占用（PERCENT_MEM）排序，快速找到资源消耗大户。按 `F5` 切换到树形视图，可以清晰看到父子进程的关系。

## lsof：列出打开的文件

在 Linux 中，一切皆文件 —— 网络连接、管道、设备都体现为文件。`lsof`（List Open Files）可以查看进程打开了哪些资源。

```bash
# 查看特定进程打开的文件
lsof -p 1234

# 查看哪个进程在使用特定端口
lsof -i :80
lsof -i :443

# 查看特定用户打开的进程
lsof -u www-data

# 查看被删除但仍被进程占用的文件（释放磁盘空间时常用）
lsof +L1

# 列出所有网络连接
lsof -i
```

常见的排障场景：

```bash
# 端口被占用，找不到是谁在监听
lsof -i :8080

# 发现磁盘满了，但删除大文件后空间未释放
# 找出被删但未释放的文件
lsof | grep deleted

# 查看进程的工作目录和可执行文件路径
lsof -p $(pgrep nginx) | grep -E 'cwd|txt'
```

## strace：系统调用追踪

`strace` 追踪进程发起的系统调用，是深入调试程序行为的利器。

```bash
# 追踪一个新进程并显示时间戳
strace -ttt -f -o /tmp/trace.log python app.py

# 附加到运行中的进程
sudo strace -p 1234

# 只追踪特定类型的系统调用
strace -e trace=open,read,write -p 1234

# 只追踪网络相关系统调用
strace -e trace=network -p 1234

# 统计每种系统调用的耗时和次数（性能分析利器）
strace -c -p 1234
# 按 Ctrl+C 停止后，会打印汇总统计表
```

典型排查场景：

```bash
# 程序提示 "No such file or directory"，想知道它在找什么文件
strace -e trace=openat,stat ./myapp 2>&1 | grep ENOENT

# 数据库连接超时，追踪网络调用
strace -e trace=connect,sendto,recvfrom -tt -p $(pgrep mysqld)
```

## ps 命令进阶用法

```bash
# 按 CPU 占用排序，显示前 10
ps aux --sort=-%cpu | head -10

# 按内存占用排序，显示前 10
ps aux --sort=-%rss | head -10

# 按进程名查找
ps aux | grep java

# 显示进程树，查看父子关系
ps auxf

# 查看所有进程的线程数量
ps -eo pid,comm,nlwp | sort -k3 -rn | head -10
```

## 进程信号管理

`kill` 命令本质上是向进程发送信号。掌握常用的信号类型，避免一味使用 `kill -9`：

| 信号编号 | 信号名 | 用途 |
|----------|--------|------|
| 1 | SIGHUP | 重新加载配置（常用于 Nginx、HAProxy） |
| 2 | SIGINT | 中断进程（等效于 Ctrl+C） |
| 9 | SIGKILL | 强制终止（不可捕获，无清理机会） |
| 15 | SIGTERM | 优雅终止（默认信号，给进程清理的机会） |
| 17 | SIGSTOP | 暂停进程（不可捕获） |
| 18 | SIGCONT | 恢复被暂停的进程 |

```bash
# 优雅终止（首选）
kill 1234

# 强制终止（进程无响应时使用）
kill -9 1234

# 重新加载配置
kill -1 $(cat /var/run/nginx.pid)
# 或使用
sudo nginx -s reload

# 暂停和恢复进程
kill -STOP 1234     # 暂停
kill -CONT 1234     # 恢复
```

## 典型排障流程

1. **CPU 飙高**：`htop` 按 CPU 排序，找到高 CPU 进程；`strace -c -p PID` 分析系统调用耗时分布；`lsof -p PID` 查看进程正在做什么
2. **端口冲突**：`lsof -i :PORT` 直接定位占用端口的进程
3. **程序假死**：先 `kill PID`（SIGTERM）尝试优雅退出；无响应再用 `kill -9`；退出后检查 `journalctl -u service-name -b -1` 查看是否有异常日志
4. **磁盘空间"消失"**：`df -h` 确认分区已满；`lsof +L1` 找出被删除但未释放的文件，重启对应进程即可释放

## 小结

`htop` 提供直观的资源监控，`lsof` 揭示文件与网络的关联关系，`strace` 深入系统调用级别的问题定位。三者配合使用，可以覆盖绝大多数服务器进程层面的故障排查需求。
