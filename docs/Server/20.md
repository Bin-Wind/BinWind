# 服务器性能压测

上线前知道系统能扛多少流量，是运维的基本素养。压力测试不是为了追求一个好看的 QPS 数字，而是找到系统的极限点、发现瓶颈所在。本文介绍 `ab`（ApacheBench）和 `wrk` 两个压测工具的使用方法和结果分析。

## ab (ApacheBench) 快速压测

`ab` 随 Apache 安装包一起发布，是入门级的 HTTP 压测工具：

```bash
# 安装
sudo apt install apache2-utils  # Debian/Ubuntu
sudo yum install httpd-tools     # CentOS/RHEL
```

基础用法：

```bash
# 发送 1000 个请求，并发 10
ab -n 1000 -c 10 https://example.com/

# 带请求头
ab -n 500 -c 20 -H "Authorization: Bearer xxx" https://api.example.com/

# POST 请求（从文件读取请求体）
ab -n 500 -c 10 -p post_data.json -T "application/json" https://api.example.com/endpoint

# 限制测试时间（替换 -n 参数）
ab -t 30 -c 10 https://example.com/

# 开启 keep-alive
ab -n 1000 -c 10 -k https://example.com/

# 设置超时（默认 30 秒）
ab -n 1000 -c 10 -s 5 https://example.com/
```

核心指标：
- **Requests per second**（RPS）：系统吞吐量，越高越好
- **Time per request**（用户视角）：单个用户的平均等待时间，越低越好
- **Failed requests**：失败请求数，压测过程中应为 0
- **P90 / P95 / P99**：百分位延迟，反映尾部延迟的真实用户体验

## wrk：高性能压测工具

`wrk` 是更现代的压测工具，支持多线程、Lua 脚本扩展，适合高并发场景。

```bash
# 安装
sudo apt install wrk          # Debian/Ubuntu
sudo yum install wrk          # CentOS/RHEL

# 或从源码编译
git clone https://github.com/wg/wrk.git
cd wrk && make
sudo cp wrk /usr/local/bin/
```

基础用法：

```bash
# 4 线程、100 并发、持续 30 秒
wrk -t4 -c100 -d30s https://example.com/

# 自定义超时
wrk -t8 -c200 -d60s --timeout 5s https://api.example.com/

# 使用脚本自定义请求
wrk -t4 -c100 -d30s -s custom.lua https://example.com/
```

## wrk Lua 脚本

```lua
-- custom.lua：发送 JSON POST 请求
wrk.method = "POST"
wrk.headers["Content-Type"] = "application/json"
wrk.headers["Authorization"] = "Bearer your-token-here"
wrk.body = '{"name": "test", "value": 123}'

-- 自定义请求生成
request = function()
    local path = "/api/users/" .. math.random(1, 1000)
    return wrk.format("GET", path)
end

-- 处理响应（记录延迟分布）
latencies = {}
response = function(status, headers, body)
    latencies[#latencies + 1] = 1
    if status ~= 200 then
        io.write("ERROR: status=" .. status .. "\n")
    end
end

-- 测试结束后输出汇总
done = function(summary)
    io.write(string.format("Total requests: %d\n", summary.requests))
    io.write(string.format("Avg latency: %.2f ms\n", summary.latency.mean / 1000))
    io.write(string.format("Requests/sec: %.2f\n",
        summary.requests / (summary.duration / 1000000)))
end
```

## 压测前准备

**1. 预热**：先以低并发请求几分钟，让 JIT 编译、缓存等生效：

```bash
wrk -t2 -c10 -d120s --latency https://api.example.com/health
```

**2. 从独立机器压测**：不要在目标服务器上运行压测工具，压测工具本身也会消耗大量 CPU 和网络资源。使用同机房的另一台服务器，或者云上的临时实例。

**3. 监控服务器资源**：压测过程中在目标服务器上运行：

```bash
# 终端一：CPU 和进程
htop

# 终端二：网络流量
sudo iftop

# 终端三：系统资源详情
vmstat 1
iostat -x 1

# 终端四：应用日志
journalctl -u myapp -f
```

**4. 确认文件描述符限制**：

```bash
# 查看当前限制
ulimit -n

# 临时提高（生产环境可能需要 65535 或更高）
sudo sysctl -w fs.file-max=65535
ulimit -n 65535
```

## 系统性压测方案

逐步提高并发，记录每个阶段的 RPS、延迟和服务器资源，找到"拐点"——性能开始显著下降的并发数值：

```bash
#!/bin/bash
# 梯度压测脚本
URL="https://api.example.com/endpoint"
CONCURRENCIES=(10 50 100 200 500 1000)
DURATION="60s"
THREADS=8

echo "并发数,RPS,P50(ms),P99(ms),错误率"
for c in "${CONCURRENCIES[@]}"; do
    echo -n "测试并发 ${c}..."
    result=$(wrk -t${THREADS} -c${c} -d${DURATION} --latency "$URL" 2>&1)
    rps=$(echo "$result" | grep "Requests/sec" | awk '{print $2}')
    p50=$(echo "$result" | grep "50%" | awk '{print $2}')
    p99=$(echo "$result" | grep "99%" | awk '{print $2}')
    errors=$(echo "$result" | grep "Non-2xx or 3xx" | awk '{print $5}')
    echo "${c},${rps},${p50},${p99},${errors:-0}"
    sleep 10       # 让系统恢复
done
```

## 常见瓶颈分析

| 症状 | 可能原因 | 排查方向 |
|------|----------|----------|
| RPS 上不去，CPU 闲置 | 网络带宽耗尽、连接数限制 | `iftop` 看流量、`ulimit -n` 看 fd 限制 |
| CPU 100%，RPS 开始下降 | 计算密集型瓶颈 | 优化代码、增加实例或 CPU 核数 |
| 错误率随并发增大而升高 | 数据库连接池耗尽、队列溢出 | 检查 DB 连接数和 wait_timeout |
| P99 远高于 P50 | 存在慢请求、GC 暂停 | 添加链路追踪，分析慢请求路径 |

## 小结

压力测试不是一次性的工作，而应该嵌入到开发流程中——每次重大变更后都重新跑一次基准测试，记录性能趋势。`ab` 轻便快速，适合简单压测；`wrk` 更强大，适合高并发场景和生产级测试。压测时务必从独立机器发起请求，并同时监控服务器资源，才能准确找到性能瓶颈。
