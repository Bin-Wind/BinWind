# DNS 配置与域名解析原理

当你在浏览器中输入网址并按下回车，背后发生的第一件事就是 DNS 解析——将人类可读的域名转换为服务器的 IP 地址。理解 DNS 的工作原理和各种记录类型，是排查网络问题和管理域名的基础能力。

## DNS 解析流程

一次完整的 DNS 查询会经历以下步骤：

```
浏览器缓存
    ↓ (未命中)
操作系统 DNS 缓存
    ↓ (未命中)
本地 DNS 解析器 (ISP 或 8.8.8.8)
    ↓ (未命中)
根域名服务器 (.)  →  返回 .com 的 TLD 服务器地址
    ↓
TLD 服务器 (.com) →  返回 example.com 的权威服务器地址
    ↓
权威 DNS 服务器   →  返回 example.com 的 A 记录（IP 地址）
    ↓
返回给浏览器，建立连接
```

简而言之：**递归查询 + 迭代查询** 的组合，层层递进直到找到权威答案。

## DNS 记录类型

| 记录类型 | 用途 | 示例 |
|----------|------|------|
| **A** | 将域名指向 IPv4 地址 | `example.com A 93.184.216.34` |
| **AAAA** | 将域名指向 IPv6 地址 | `example.com AAAA 2606:2800:220:1:248:1893:25c8:1946` |
| **CNAME** | 域名别名（指向另一个域名） | `www.example.com CNAME example.com` |
| **MX** | 邮件服务器 | `example.com MX 10 mail.example.com` |
| **TXT** | 文本记录（常用于 SPF/DKIM 验证） | `example.com TXT "v=spf1 include:_spf.google.com ~all"` |
| **NS** | 指定该域的权威 DNS 服务器 | `example.com NS ns1.dnsprovider.com` |
| **SOA** | 域的起始授权记录（管理信息） | 包含主 NS、管理员邮箱、序列号、TTL 等 |

## dig 命令详解

`dig` 是 DNS 调试的瑞士军刀，输出详细且标准化。

```bash
# 基本查询
dig example.com

# 简洁输出
dig example.com +short      # 只显示 IP
dig example.com +noall +answer   # 只显示答案

# 指定记录类型
dig example.com A
dig example.com AAAA
dig example.com MX           # 查看邮件服务器
dig example.com TXT          # 查看 TXT 记录（常用于验证域名所有权）
dig example.com NS           # 查看权威 DNS 服务器
dig example.com ANY          # 查询所有记录类型

# 指定 DNS 服务器查询（绕过本地缓存）
dig @8.8.8.8 example.com
dig @1.1.1.1 example.com

# 追踪完整解析路径
dig example.com +trace

# 反向查询（IP → 域名）
dig -x 93.184.216.34

# 查看 DNS 响应的 TTL（剩余缓存时间）
dig example.com | grep "ANSWER SECTION" -A 1
# 答案中的第二个字段即为 TTL（秒）
```

## nslookup 基础用法

`nslookup` 是最简单的 DNS 查询工具，适合快速验证：

```bash
# 基本查询
nslookup example.com

# 指定 DNS 服务器
nslookup example.com 8.8.8.8

# 查询 MX 记录
nslookup -type=MX example.com

# 反向查询
nslookup 93.184.216.34
```

## 常见域名配置

### A 记录：核心域名解析

```text
@     A    93.184.216.34      # 根域名 example.com 解析
www   A    93.184.216.34      # www 子域名解析
api   A    10.0.1.5            # API 子域名指向内网 IP
```

### CNAME：别名记录

```text
www       CNAME   example.com           # www 指向根域名
assets    CNAME   cdn.example.net       # 静态资源指向 CDN
shop      CNAME   stores.shopify.com    # 电商子域名指向 Shopify
```

注意：**根域名（@）不能使用 CNAME 记录**（根据 RFC 规范，CNAME 不能与其他记录共存）。对于根域名的 CDN 接入，应使用 A 记录指向 CDN 提供的 IP，或使用 CDN 提供商支持的别名方式（如 Cloudflare 的 CNAME Flattening、ALIAS/ANAME 记录）。

### MX：邮件记录

```text
@   MX   10    mail.example.com      # 主邮件服务器，优先级 10
@   MX   20    backup-mail.example.com   # 备用，优先级 20（数字越小优先级越高）
```

### TXT：验证与安全

```text
# SPF：指定哪些服务器可以代表该域发送邮件
@   TXT   "v=spf1 include:_spf.google.com ~all"

# DKIM：邮件签名验证
google._domainkey   TXT   "v=DKIM1; k=rsa; p=MIGfMA0GCSq..."

# DMARC：邮件验证策略
_dmarc   TXT   "v=DMARC1; p=quarantine; rua=mailto:dmarc@example.com"

# 域名所有权验证（Let's Encrypt、GitHub Pages 等服务使用）
_acme-challenge   TXT   "random-verification-string"
```

## DNS 传播与 TTL

修改 DNS 记录后，并非立即全球生效。关键在于 **TTL（Time to Live）** 值，它决定了各级 DNS 服务器缓存该记录的时间。

```bash
# 查看当前记录的 TTL
dig example.com +noall +answer

# 输出示例：
# example.com.  300  IN  A  93.184.216.34
#              ↑ TTL = 300 秒（5分钟）
```

变更 DNS 的最佳实践：
- **变更前先降低 TTL**：提前将 TTL 从 3600（1小时）降至 300（5分钟），等待旧 TTL 过期后再修改记录
- **修改后恢复 TTL**：确认解析生效后，将 TTL 调回原值
- **验证传播状态**：使用 `dig @8.8.8.8`、`dig @1.1.1.1` 分别查询不同 DNS 服务器，确认新 IP 已生效。或者使用在线工具如 whatsmydns.net 查看全球范围内的解析情况

## DNS 缓存清理

排查 DNS 问题时，先确保本地缓存不是过期的：

```bash
# macOS
sudo dscacheutil -flushcache
sudo killall -HUP mDNSResponder

# Linux (systemd-resolved)
sudo systemd-resolve --flush-caches

# Linux (nscd)
sudo systemctl restart nscd

# Windows
ipconfig /flushdns

# Chrome 浏览器内部 DNS 缓存
# 访问 chrome://net-internals/#dns 点击 "Clear host cache"
```

## /etc/hosts 本地 DNS 覆盖

`/etc/hosts` 文件可以手动指定域名到 IP 的映射，优先级高于 DNS 查询。

```bash
# /etc/hosts
127.0.0.1   localhost
192.168.1.100   app.local
::1             ip6-localhost

# 临时屏蔽某个域名
0.0.0.0   annoying-ad-server.com

# 测试新服务器（DNS 还未切时）
203.0.113.10   api.example.com
```

这在以下场景非常有用：
- 本地开发时模拟生产环境域名
- 在 DNS 切换前提前验证新服务器
- 临时屏蔽不必要的网络请求

## 常见 DNS 故障排查

```bash
# 1. 确认域名能否解析
dig example.com +short
# 若无输出，说明 DNS 解析失败

# 2. 换个 DNS 服务器确认是本地 DNS 问题还是权威 DNS 问题
dig @8.8.8.8 example.com
dig @1.1.1.1 example.com

# 3. 追踪解析路径定位断点
dig example.com +trace

# 4. 检查域名注册状态和 NS 服务器配置
whois example.com | grep -E "Name Server|Expir"

# 5. 检查权威 NS 的一致性（所有 NS 应返回相同结果）
# 先查 NS 列表
dig example.com NS +short
# 再逐个 NS 查询
dig @ns1.provider.com example.com +short
dig @ns2.provider.com example.com +short
```

## 小结

DNS 是互联网的基础设施，理解解析流程和记录类型对域名管理和故障排查至关重要。`dig` 命令是排障的核心工具，`+trace` 追踪解析路径，`+short` 快速获取答案。关键记录配置时注意 A 记录与 CNAME 的兼容限制，变更前提前降低 TTL 能大大缩短切换时间。
