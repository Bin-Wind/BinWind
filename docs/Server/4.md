# Nginx 反向代理配置实战

Nginx 是目前最流行的高性能 Web 服务器和反向代理服务器。作为反向代理，它可以将客户端请求转发到后端应用服务器，实现负载均衡、缓存、SSL 终结等功能。本文将从安装开始，逐步深入 Nginx 反向代理的配置实践。

## 安装 Nginx

```bash
# Linux (Debian/Ubuntu)
sudo apt update
sudo apt install nginx

# Linux (CentOS/RHEL)
sudo yum install nginx

# macOS
brew install nginx

# 验证安装
nginx -v
```

安装后启动并设置开机自启：
```bash
sudo systemctl start nginx
sudo systemctl enable nginx
```

## Nginx 配置文件结构

Nginx 的主配置文件位于 `/etc/nginx/nginx.conf`，典型结构如下：

```nginx
# /etc/nginx/nginx.conf
user www-data;
worker_processes auto;
pid /run/nginx.pid;

events {
    worker_connections 1024;
    use epoll;
}

http {
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    # 日志格式
    log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer" '
                    '"$http_user_agent" "$http_x_forwarded_for"';

    access_log /var/log/nginx/access.log main;
    error_log /var/log/nginx/error.log warn;

    sendfile on;
    tcp_nopush on;
    keepalive_timeout 65;

    # 引入站点配置
    include /etc/nginx/conf.d/*.conf;
    include /etc/nginx/sites-enabled/*;
}
```

推荐将每个站点的配置独立放在 `/etc/nginx/sites-available/` 目录下，然后软链接到 `sites-enabled/` 以启用：

```bash
sudo ln -s /etc/nginx/sites-available/mysite.conf /etc/nginx/sites-enabled/
```

## 基础反向代理配置

最基本的反向代理：将来自 80 端口的请求转发到本地 `localhost:3000` 的 Node.js 应用：

```nginx
server {
    listen 80;
    server_name example.com www.example.com;

    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

## 常用代理头说明

反向代理必须正确设置请求头，确保后端应用能获取到真实的客户端信息：

| 头部 | 说明 |
|------|------|
| `Host` | 客户端请求的原始主机名 |
| `X-Real-IP` | 客户端真实 IP |
| `X-Forwarded-For` | 代理链中的客户端 IP 列表 |
| `X-Forwarded-Proto` | 客户端原始协议（http/https） |
| `X-Forwarded-Host` | 客户端请求的原始主机名（多级代理场景） |

常见配置模板：

```nginx
location / {
    proxy_pass http://backend;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header X-Forwarded-Host $host;

    # 超时设置（WebSocket 等长连接需要足够长）
    proxy_read_timeout 60s;
    proxy_send_timeout 60s;
    proxy_connect_timeout 10s;
}
```

`proxy_http_version 1.1` 和 `Connection "upgrade"` 配置是 WebSocket 支持的关键。

## 负载均衡配置

Nginx 通过 `upstream` 模块实现后端服务器负载均衡：

### 轮询（Round Robin，默认）
```nginx
upstream app_servers {
    server 10.0.0.1:3000;
    server 10.0.0.2:3000;
    server 10.0.0.3:3000;
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://app_servers;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

### 带权重的负载均衡
```nginx
upstream app_servers {
    server 10.0.0.1:3000 weight=3;  # 处理 3/6 的请求
    server 10.0.0.2:3000 weight=2;  # 处理 2/6 的请求
    server 10.0.0.3:3000 weight=1;  # 处理 1/6 的请求
}
```

### 最小连接数（least_conn）
```nginx
upstream app_servers {
    least_conn;
    server 10.0.0.1:3000;
    server 10.0.0.2:3000;
    server 10.0.0.3:3000;
}
```

### IP 哈希（会话保持）
```nginx
upstream app_servers {
    ip_hash;
    server 10.0.0.1:3000;
    server 10.0.0.2:3000;
    server 10.0.0.3:3000;
}
```

`ip_hash` 确保同一客户端 IP 始终路由到同一后端服务器，适用于需要会话保持的应用。

### 健康检查
```nginx
upstream app_servers {
    server 10.0.0.1:3000 max_fails=3 fail_timeout=30s;
    server 10.0.0.2:3000 max_fails=3 fail_timeout=30s;
    server 10.0.0.3:3000 backup;  # 备用服务器，仅在主服务器全部故障时使用
}
```

- `max_fails`：在 `fail_timeout` 时间内允许的最大失败次数
- `fail_timeout`：失败后暂停使用该服务器的时间
- `backup`：标记为备用服务器

## 代理缓存配置

Nginx 可以缓存后端响应，减轻后端压力并提升响应速度：

```nginx
# http 块中定义缓存路径
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=my_cache:10m
                 max_size=1g inactive=60m use_temp_path=off;

server {
    listen 80;
    server_name example.com;

    location /api/ {
        proxy_pass http://app_servers;
        proxy_cache my_cache;
        proxy_cache_valid 200 302 10m;   # 200/302 响应缓存 10 分钟
        proxy_cache_valid 404 1m;         # 404 响应缓存 1 分钟
        proxy_cache_key "$scheme$request_method$host$request_uri";
        proxy_cache_bypass $http_cache_control;  # 客户端请求刷新时绕过缓存
        add_header X-Cache-Status $upstream_cache_status;  # 响应头显示缓存状态
    }
}
```

## Reload vs Restart

配置修改后需要让 Nginx 重新加载，两个命令有本质区别：

```bash
# 重载配置，不中断现有连接（推荐）
sudo nginx -t              # 先测试配置文件语法
sudo nginx -s reload       # 或：sudo systemctl reload nginx

# 完全重启，会中断现有连接（生产环境慎用）
sudo systemctl restart nginx
```

**最佳实践**：永远先执行 `nginx -t` 验证配置，确认无误后再 reload。如果 `nginx -t` 报错，reload 会失败，但 Nginx 会继续运行旧配置，不会导致服务中断。
