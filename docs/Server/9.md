# MySQL 备份与恢复策略

数据是业务的核心资产，备份是防范数据丢失的最后一道防线。MySQL 提供了多种备份方式，本文将系统介绍逻辑备份、二进制日志恢复、物理备份以及自动化备份方案，帮助你建立可靠的数据保护策略。

## 备份策略概述

| 备份类型 | 工具 | 优势 | 劣势 |
|----------|------|------|------|
| 逻辑备份 | mysqldump | 简单、可读、跨版本兼容 | 大库备份恢复慢 |
| 物理备份 | XtraBackup | 速度快、在线热备 | 跨平台兼容性差 |
| 二进制日志 | binlog | 支持增量恢复/时间点恢复 | 需配合全量备份使用 |

理想的备份策略：**每周全量备份 + 每日增量备份（binlog）+ 定期恢复演练**。

## 逻辑备份：mysqldump

mysqldump 是 MySQL 自带的逻辑备份工具，将数据库结构和数据导出为 SQL 语句文件。

### 备份单个数据库

```bash
# 备份单个数据库
mysqldump -u root -p myapp > myapp_backup.sql

# 备份时指定字符集
mysqldump -u root -p --default-character-set=utf8mb4 myapp > myapp_backup.sql

# 只备份表结构，不含数据
mysqldump -u root -p --no-data myapp > myapp_schema.sql

# 只备份数据，不含建表语句
mysqldump -u root -p --no-create-info myapp > myapp_data.sql
```

### 备份所有数据库

```bash
# 备份所有数据库
mysqldump -u root -p --all-databases > all_databases.sql

# 备份多个指定数据库
mysqldump -u root -p --databases db1 db2 db3 > multi_db.sql
```

### 生产级备份参数（推荐）

```bash
mysqldump -u root -p \
    --single-transaction \       # 不锁表，用于 InnoDB 的一致性备份
    --routines \                 # 包含存储过程和函数
    --triggers \                 # 包含触发器
    --events \                   # 包含事件调度器
    --hex-blob \                 # 二进制字段以十六进制导出
    --master-data=2 \            # 记录 binlog 位置（注释形式）
    --set-gtid-purged=OFF \      # GTID 模式下避免冲突
    --compress \                 # 压缩协议传输
    myapp | gzip > myapp_$(date +%Y%m%d).sql.gz
```

**参数详解**：
- `--single-transaction`：开工事务并设置隔离级别为 REPEATABLE READ，保证备份过程中数据一致，且不阻塞读写操作。仅对 InnoDB 表有效
- `--master-data=2`：在导出文件中以注释形式记录当前的 binlog 文件名和位置，用于后续时间点恢复
- `--routines` 和 `--triggers`：很多备份会遗漏存储过程和触发器，务必带上

### 压缩与加密

```bash
# gzip 压缩
mysqldump -u root -p myapp | gzip > myapp_backup.sql.gz

# 加密备份文件
mysqldump -u root -p myapp | gzip | \
    openssl enc -aes-256-cbc -salt -pass pass:encryption_key \
    > myapp_backup.sql.gz.enc
```

## 从备份恢复

### 恢复单库

```bash
# 从 SQL 文件恢复
mysql -u root -p myapp < myapp_backup.sql

# 从压缩文件恢复
gunzip < myapp_backup.sql.gz | mysql -u root -p myapp

# 恢复加密备份
openssl enc -aes-256-cbc -d -pass pass:encryption_key \
    -in myapp_backup.sql.gz.enc | gunzip | mysql -u root -p myapp
```

### 恢复到不同名称的数据库

```bash
# 先创建新库
mysql -u root -p -e "CREATE DATABASE myapp_restore_test CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"

# 恢复
mysql -u root -p myapp_restore_test < myapp_backup.sql
```

### 选择性恢复（恢复特定表）

```bash
# 如果备份文件较大，可从全库备份中提取特定表
sed -n '/^-- Table structure for table `users`/,/^-- Table structure for table/p' \
    full_backup.sql | mysql -u root -p myapp
```

## 二进制日志（Binlog）与时间点恢复

Binlog 记录了所有对数据库的修改操作，是实现时间点恢复（PITR）的关键。

### 开启 Binlog

```ini
# my.cnf
[mysqld]
server-id = 1
log_bin = /var/log/mysql/mysql-bin
binlog_format = ROW               # ROW 格式记录每行变化，最安全
binlog_expire_logs_seconds = 604800  # 7 天自动清理
max_binlog_size = 100M
```

### 查看 Binlog

```bash
# 查看 binlog 文件列表
mysql -u root -p -e "SHOW BINARY LOGS;"

# 查看 binlog 内容
mysqlbinlog --base64-output=DECODE-ROWS -v /var/log/mysql/mysql-bin.000001

# 按时间范围查看
mysqlbinlog --start-datetime="2024-01-01 10:00:00" \
            --stop-datetime="2024-01-01 11:00:00" \
            /var/log/mysql/mysql-bin.000001

# 按位置查看
mysqlbinlog --start-position=107 --stop-position=2048 \
            /var/log/mysql/mysql-bin.000001
```

### 时间点恢复流程

假设上午 10:00 做了全量备份，11:30 有人误删了一张表。恢复目标：恢复到 11:29 的状态。

```bash
# 1. 先恢复全量备份（恢复到 10:00 的状态）
mysql -u root -p myapp < backup_1000.sql

# 2. 应用从备份时间到误操作前一刻的 binlog
mysqlbinlog --start-datetime="2024-01-01 10:00:00" \
            --stop-datetime="2024-01-01 11:29:00" \
            /var/log/mysql/mysql-bin.000001 \
            /var/log/mysql/mysql-bin.000002 \
            | mysql -u root -p
```

## 物理备份：Percona XtraBackup 概述

对于大型数据库，mysqldump 的备份恢复速度可能难以接受。Percona XtraBackup 是开源的 MySQL 物理热备工具，可以在不锁表的情况下在线备份。

```bash
# 安装
sudo apt install percona-xtrabackup-80  # MySQL 8.0

# 全量备份
xtrabackup --backup \
    --target-dir=/backup/mysql/full \
    --user=root --password='password'

# 准备备份（应用未完成的事务日志）
xtrabackup --prepare --target-dir=/backup/mysql/full

# 恢复
sudo systemctl stop mysql
sudo rm -rf /var/lib/mysql/*
xtrabackup --copy-back --target-dir=/backup/mysql/full
sudo chown -R mysql:mysql /var/lib/mysql
sudo systemctl start mysql

# 增量备份（基于上一次全量备份）
xtrabackup --backup \
    --target-dir=/backup/mysql/inc1 \
    --incremental-basedir=/backup/mysql/full \
    --user=root --password='password'
```

XtraBackup 适合数据库大于 10GB 的场景。对于百 GB 级别的数据库，这是唯一可行的在线备份方案。

## 自动化备份脚本

将备份任务配置为 cron 定时执行：

```bash
#!/bin/bash
# /usr/local/bin/mysql_backup.sh
# 每日自动备份 MySQL 数据库

BACKUP_DIR="/backup/mysql"
RETENTION_DAYS=7              # 保留最近 7 天的备份
MYSQL_USER="backup"
MYSQL_PASS="backup_password"
DATE=$(date +%Y%m%d_%H%M%S)
LOG_FILE="/var/log/mysql_backup.log"

mkdir -p $BACKUP_DIR

# 备份所有数据库
mysqldump -u $MYSQL_USER -p$MYSQL_PASS \
    --single-transaction \
    --routines \
    --triggers \
    --events \
    --all-databases \
    | gzip > $BACKUP_DIR/all_db_${DATE}.sql.gz

# 记录备份结果
if [ $? -eq 0 ]; then
    echo "[$(date)] 备份成功: all_db_${DATE}.sql.gz" >> $LOG_FILE
else
    echo "[$(date)] 备份失败!" >> $LOG_FILE
fi

# 清理过期备份
find $BACKUP_DIR -name "all_db_*.sql.gz" -mtime +$RETENTION_DAYS -delete
echo "[$(date)] 已清理 ${RETENTION_DAYS} 天前的备份" >> $LOG_FILE
```

```bash
# 配置 cron 任务（每日凌晨 2 点执行）
sudo crontab -e
0 2 * * * /usr/local/bin/mysql_backup.sh
```

## 备份验证

备份不等于安全，可恢复的备份才是真正的备份。建议定期执行恢复演练：

```bash
# 1. 启动一个临时 MySQL 实例（如 Docker 容器）
docker run -d --name mysql-restore-test \
    -e MYSQL_ROOT_PASSWORD=test123 \
    -p 3307:3306 mysql:8.0

# 2. 恢复备份到测试实例
gunzip < /backup/mysql/all_db_20240101.sql.gz | \
    mysql -h 127.0.0.1 -P 3307 -u root -ptest123

# 3. 验证数据完整性
mysql -h 127.0.0.1 -P 3307 -u root -ptest123 -e "
    SELECT table_schema, COUNT(*) AS table_count
    FROM information_schema.tables
    GROUP BY table_schema;
    SELECT table_schema, SUM(table_rows) AS total_rows
    FROM information_schema.tables
    GROUP BY table_schema;
"

# 4. 清理测试实例
docker stop mysql-restore-test && docker rm mysql-restore-test
```

## 备份策略对比与建议

| 场景 | 推荐方案 | 频率 |
|------|----------|------|
| 小型项目（< 1GB） | mysqldump 全量 | 每天一次 |
| 中型项目（1-50GB） | mysqldump + binlog | 每天全量 + binlog 持续 |
| 大型项目（> 50GB） | XtraBackup 全量 + 增量 + binlog | 每周全量 + 每日增量 |

**黄金法则**：
1. 备份存储在不同物理介质（异地、不同服务器）
2. 备份文件必须加密
3. 定期执行恢复演练（至少每季度一次）
4. 监控备份任务，确保失败时能及时告警
